# 运动生涯记录中心 V2（多运动纪录）开发任务清单

## 0. 使用说明

本文档是记录中心 V2 后续任务循环的唯一顺序依据。产品语义、运动规则、数据边界和发布门禁以 `docs/运动生涯记录中心V2（多运动纪录）开发交付手册.md` 为准。

V1 已完成的跑步事实和测试是 V2 的继承基线。原 V1 清单中的 RC-28 至 RC-30 不再代表最终发布完成；V2 完成后必须重新执行本清单的平台与发布门禁。

### 0.1 任务循环协议

每个任务必须严格执行以下循环：

1. 按清单顺序选择第一个依赖已完成的 `Pending` 任务，并将其标记为 `In Progress`。
2. 在修改代码前，先把任务条目改写为工程级执行提示词。
3. 提示词必须明确：任务目标、输入摘要、前置依赖、文件范围、冻结契约、输入输出、非目标、实施步骤、测试、差异复核和完成报告。
4. 提示词完成后立即执行，不因普通实现细节向用户逐项请求确认。
5. 运行与风险相称的定向测试、契约检查和必要宽回归。
6. 对实际 diff 做自适应复核，重点检查越界计算、错误事实源、未授权真实库写入、敏感信息泄露和无关文件修改。
7. 测试失败时在任务范围内自修复；连续失败或需要改变冻结产品语义时停止并请求用户判断。
8. 任务通过后新增独立完成报告、刷新滚动摘要、更新本清单状态，再进入下一任务。
9. 自动化测试通过不等于发布完成；真实数据、视觉、macOS、Windows 和打包分别验收。
10. 每次只允许一个主任务处于 `In Progress`。

### 0.2 首任务与滚动摘要协议

`RCV2-00` 必须全文阅读并形成摘要：

- V2 开发交付手册。
- V1 开发交付手册。
- V1 与 V2 任务清单。
- `docs/js_api_contract.json` 中 Career/Records 契约。
- `docs/records_center_rolling_contract_summary.md`。
- 当前代码、测试、工作树和真实数据审计状态。

`RCV2-00` 创建：

```text
docs/records_center_v2_rolling_contract_summary.md
```

从 `RCV2-01` 开始，每个任务默认只需阅读：

1. V2 滚动摘要。
2. 当前任务条目。
3. 上一任务完成报告。
4. 当前任务直接涉及的代码、契约和测试。

每个任务结束后必须刷新滚动摘要。只有出现以下情况才全文重读交付手册和契约：

- 实现与摘要严重冲突。
- 需要改变已冻结运动规则、Scope、比较方向或状态语义。
- API/schema 出现无法由兼容扩展解决的冲突。
- 真实数据结果与规则预期出现系统性偏离。
- 滚动摘要缺失关键事实，无法安全继续。

### 0.3 状态标记

- `Pending`：尚未开始。
- `In Progress`：正在执行，最多一个。
- `Blocked`：需要用户决策或外部环境才能继续。
- `Done`：交付物、定向测试、diff 复核和完成报告均满足。
- `Verified`：在后续真实数据或平台门禁中再次验证通过。

### 0.4 全局硬约束

所有任务必须遵守：

- Activity 是唯一事实源。
- Resolver/状态迁移服务是正式纪录唯一写入口。
- 前端只渲染 ViewModel，不计算纪录、Scope、累计提升或置信度。
- AI 只消费安全 Snapshot，不读取 raw FIT、功率流、轨迹、路径、schema 或体重历史。
- 每条纪录必须绑定 `activity_id`；分段纪录必须绑定 Activity 内范围。
- `career_pb_records`、`career_record_events` 和候选状态机保持兼容，不新建重叠事实表。
- `get_career_pb*` 和 `detail_link.source = "career"` 保持兼容。
- `>0.90` 自动确认，`0.70-0.90` 候选，`<0.70` 忽略；边界 `0.70/0.90` 均为候选。
- 跑步 V1 四项结果不得因 V2 改变。
- 模型估计和分析曲线不得污染正式纪录。
- 未通过数据验收的运动必须在 Catalog 中 `available=false`。
- 第一次 V2 真实数据评估只允许备份、staging 和 dry-run；没有用户批准不得 apply 真实库。
- 用户仍未授权重新打包；RCV2-44 前不得生成、签名、公证或替换发布包。
- 工作区已有改动属于用户；不得覆盖、回退或整理无关修改。

- V2 滚动摘要已建立。

- Registry、Scope、质量、schema、API、视觉和发布门禁已冻结。

## Milestone B：通用 Records 底座

包含：`RCV2-09` 至 `RCV2-14`。

通过条件：

- V2 Catalog、Scope、Evidence、状态机、增量/重建和通用 API 可用。
- V1 跑步纪录和旧 PB API 无回归。
- Curve/Route cache 明确不是 canonical record。

## Milestone C：多运动 Resolver

包含：`RCV2-15` 至 `RCV2-31`。

通过条件：

- 骑行、徒步、游泳和越野按各自规则生成事实或 candidate-only 结果。
- 没有样本或未验收的运动在 Catalog 中不可用。
- 模型和分析曲线未写入正式纪录。

## Milestone D：统一前端体验

包含：`RCV2-32` 至 `RCV2-35`。

通过条件：

- 新视觉在现有脉图外壳中完成。

- 当前、演进、候选以及全部状态在桌面和窄窗口可用。
- 前端没有表现指数、伪数据或业务推断。

## Milestone E：跨模块与质量闭环

包含：`RCV2-36` 至 `RCV2-39`。

通过条件：

- Overview、Timeline、Achievement、Snapshot、AI、Trends 边界正确。
- 安全、性能、日志、可观测性和宽回归通过。

## Milestone F：真实数据与用户决策

包含：`RCV2-40` 至 `RCV2-41`。

## Milestone G：平台与发布

包含：`RCV2-42` 至 `RCV2-44`。

通过条件：

- macOS、Windows 当前代码和视觉均验收。

# 3. 详细任务

## RCV2-00：V2 基线审计与滚动摘要初始化

任务目标：建立 V2 唯一可信起点，让后续任务不再反复猜测 V1 状态、工作树边界和 V2 决策。

前置条件：无。

必须完成：

- 全文阅读 0.2 节规定的手册、清单、契约和滚动摘要。
- 审计 `career_backend.py`、`main.py`、`track.html`、`docs/js_api_contract.json` 与相关测试的真实状态。
- 区分 V1 已完成事实、V2 目标、遗留发布风险和无关工作树改动。
- 记录真实库“候选保留、不写入”和“暂不打包”的用户决策。
- 创建 `docs/records_center_v2_rolling_contract_summary.md`。
- 为本任务写工程级提示词、基线测试记录和完成报告。

明确非目标：不改业务代码、不写数据库、不调整 UI、不打包。

交付物：V2 滚动摘要、基线审计报告、基线测试结果。

完成标准：后续任务只读滚动摘要即可准确复述事实源、规则、任务顺序、当前风险和文件边界。

建议验证：V1 Career 定向测试、Python compile、API JSON 校验、工作树差异清单。

## RCV2-01：多运动 Activity 事实源与真实数据审计

任务目标：确认骑行、徒步、游泳和越野所需 canonical 字段、时间语义、采样完整度和真实数据覆盖。

前置条件：`RCV2-00`。

必须完成：

- 画出 FIT/GPX/provider 到 Activity 的距离、时间、功率、海拔、Lap、泳姿、pool length、轨迹来源链。
- 验证 elapsed/timer/moving time 的字段语义和旧数据兼容性。
- 统计真实库各运动 Activity 数量、字段覆盖、功率流、Lap、轨迹和海拔质量。
- 审计历史体重是否可按活动日期可靠关联。
- 标记无真实样本的泳池和越野能力。
- 回答 V2 手册附录 B 中与事实源直接相关的问题。

契约边界：只读审计；不得从文件名或标题直接提升运动类型置信度；不得写真实库。

交付物：多运动事实源审计、字段矩阵、真实数据覆盖报告。

完成标准：每个 V2 Registry 定义都能指向明确 canonical 输入或被标记不可实现/候选。

建议验证：SQLite 只读查询、FIT fixture 解析测试、Activity import/sync 定向回归。

## RCV2-02：Golden fixtures 与算法可行性样本冻结

任务目标：建立可复算的最小样本集，覆盖真实库缺失的泳池、越野、功率断点和异常海拔场景。

前置条件：`RCV2-01`。

必须完成：

- 选择或构造不含个人敏感信息的骑行功率、徒步海拔、泳池 Length/Lap、公开水域和越野路线 fixtures。
- 为每个 fixture 冻结输入事实、预期证据和不可用原因。
- 覆盖功率 0W、缺失、非 1Hz、暂停、尖峰；泳池休息/泳姿；路线同向/反向/低重合。
- 定义 golden output 版本和更新流程。

契约边界：fixture 不得来自未经脱敏的真实轨迹；不得把测试预期反向当作产品规则。

交付物：fixtures、golden manifest、可行性报告、测试 helper。

完成标准：后续 Resolver 可以在无真实泳池/越野样本时得到确定且可审计的测试输入。

建议验证：fixture schema 检查、重复运行一致性、敏感字段扫描。

## RCV2-03：V2 Record Registry、纪录族与 Scope 冻结

任务目标：把手册中的跑步、骑行、徒步、游泳和越野定义变成无歧义 Registry 契约。

前置条件：`RCV2-01`、`RCV2-02`。

必须完成：

- 冻结 `RecordDefinition` V2 字段、纪录族、单位、比较方向、source_mode 和 priority。
- 冻结 power metric、indoor、pool length、stroke、water、route、segment 等 Scope。
- 列出静态 record keys 与动态 `record_key + scope_key` 规则。
- 冻结哪些定义 V2 首发 available，哪些 candidate-only 或 unavailable。
- 定义 Registry 版本兼容和禁用策略。

契约边界：不改变跑步四项定义；不把模型估计注册为正式纪录。

交付物：Registry 契约、定义表、冲突矩阵、计划测试。

完成标准：新增任何纪录不需要开发者自行猜测值、单位、方向、Scope 或来源模式。

建议验证：定义唯一性、单位/方向/source_mode 合法性、Scope 冲突测试设计。

## RCV2-04：质量评分、置信度与原因码冻结

任务目标：统一多运动的自动确认、候选和忽略边界，并保证原因可解释。

前置条件：`RCV2-03`。

必须完成：

- 保留 V1 置信度阈值和边界行为。
- 冻结功率尖峰/断点、体重缺失、海拔质量、pool length、泳姿、GPS、路线重合等评分维度。
- 冻结 V2 reason codes、用户文案和日志安全映射。

## RCV2-26：游泳正式纪录、Catalog、API 与测试闭环

# 4. 每任务工程提示词模板

每次任务循环生成的工程级提示词至少包含：

```text
任务：RCV2-XX <名称>

一、任务目标
二、滚动摘要与上一任务输入
三、前置依赖和已冻结契约
四、允许修改的文件/模块
五、禁止修改和明确非目标
六、输入/输出、API、schema、状态和错误契约
七、实施步骤
八、定向测试和升级回归条件
九、自适应 diff 复核清单
十、完成报告、滚动摘要和任务状态更新
```

提示词必须包含以下停止条件：

- 需要改变交付手册的产品语义。
- 需要未经授权写真实库或打包。
- 需要覆盖无关工作树修改。
- 同一阻塞连续复现且无法在任务边界内解决。

10. 手册、任务清单、代码、API contract、测试、发布说明和回滚说明一致。
