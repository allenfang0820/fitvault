# 运动生涯记录中心 V2（多运动纪录）开发交付手册

> 本手册是记录中心 V2 的产品、数据、Resolver、API、前端、AI、测试和发布统一依据。
> V2 继承 V1 已完成的跑步 PB 事实，不重写 V1 判定规则；新增骑行、徒步、游泳和越野跑纪录族，并完成记录中心视觉重构。
> 若代码、任务清单或实现提示词与本文冲突，必须先更新本文和接口契约，不得由前端、AI 或单个 Resolver 静默改变纪录语义。

V2 的有序开发任务应在本文冻结后单独生成。现有 V1 任务清单中的发布与打包任务不得在 V2 视觉和多运动验收完成前被视为最终发布门禁。

---

# 0. 文档结论

## 0.1 一句话定义

记录中心 V2 用统一视觉框架展示不同运动各自公平、可追溯的个人纪录：跑步看标准距离成绩，骑行看功率持续时间，徒步看整次活动成就，游泳区分泳池与公开水域，越野跑以整次活动和同路线/同赛段 PR 为核心。

## 0.2 产品位置与命名

- 用户可见模块名称继续使用“记录中心”。
- 运动生涯二级入口继续使用“记录”。
- “PB”只用于确实表示个人最佳成绩的纪录类型。
- “纪录”表示被创造或刷新的成绩；“记录”表示保存、管理和产品模块。
- V2 仍位于“运动生涯”内部，不升级为应用全局一级导航。

## 0.3 V2 核心交付

1. 保留跑步 5K、10K、半马、马拉松 V1 全部行为。
2. 新增骑行功率持续时间纪录及骑行整次活动纪录。
3. 新增徒步整次活动纪录。
4. 新增泳池游泳和公开水域游泳独立纪录 Scope。
5. 新增越野跑整次活动纪录、路线/赛段 PR 设计和 Pace/GAP 分析边界。
6. 使用统一的“当前纪录｜演进｜候选”信息架构和多运动视觉系统。
7. 所有正式纪录继续由后端 Resolver 生成，前端不计算纪录事实。
8. 所有新纪录都能回溯到 Activity；分段纪录还必须回溯到活动内起止范围。
9. 旧数据、设备异常和口径不明结果默认候选或忽略，不污染正式纪录。
10. V2 发布前完成真实数据 dry-run、人工复核、桌面/窄窗口视觉验收和宽回归。

## 0.4 V2 明确不做

- 用户手动填写或修改纪录值。
- 用户之间的公开排行榜、KOM/QOM 或年龄组排行。
- 直接复制 Strava、Intervals.icu、TrainingPeaks 的私有算法或界面。
- 把 eFTP、CP、W′、MAP、GAP、NGP、TSS 等模型输出当作未经说明的正式 PB。
- 用当前体重反推历史骑行 W/kg。
- 把不同泳池长度、泳姿、公开水域和泳池成绩混排。
- 把公路跑、越野跑、徒步、步行和登山混排。
- 在没有路线匹配证据时声称跨路线越野配速可公平比较。
- 因 V2 扩展而新建与 `career_pb_records` 含义重叠的平行事实表。
- 在用户再次授权前写入真实库、重新打包、签名或公证。

# 1. V1 继承与当前基线

## 1.1 必须继承的稳定契约

- Activity 是唯一事实源。
- Resolver 负责运动归一化、纪录匹配、质量评分、比较和状态迁移。
- `career_pb_records` 继续保存正式当前/历史纪录。
- `career_record_events` 继续保存不可变的纪录状态事件。
- `career_event_candidates` 继续保存中置信度候选。
- `get_career_pb()`、`get_career_pb_detail()`、`get_career_pb_history()` 保持兼容。
- `detail_link.source = "career"` 保持不变。
- 当前、历史、候选都必须绑定有效 `activity_id`。
- 置信度阈值继续使用：`>0.90` 自动确认，`0.70-0.90` 候选，`<0.70` 忽略。
- `0.70` 和 `0.90` 都属于候选，不自动确认。
- 前端、AI、Overview、Timeline 和 Trends 不得自行重算纪录。

## 1.2 当前代码事实

截至 2026-07-14：

- 跑步四项 V1 Registry、Resolver、状态机、API、候选和前端路径已完成。
- `get_career_pb_history()` 已能按日期升序返回 active/superseded/invalidated 历史链。
- ECharts 已作为应用本地依赖加载，可用于 V2 图表，不需要墨刀或外部 CDN。
- Career Overview 已能累计跑步、骑行、步行、徒步和游泳距离。
- FIT 导入已保存骑行活动汇总功率以及部分活动的逐点功率流。
- FIT/Resolver 已能识别泳池/公开水域语义、SWOLF、划水距离和部分 Lap 信息。
- FIT/Resolver 已能读取徒步/登山距离、累计爬升、最高海拔和轨迹。
- 当前记录中心仍残留未交付骑行筛选项；V2 之前不得把这些选项解释为已完成算法。

# 2. V2 架构契约

## 2.1 单向数据流

```text
FIT / GPX / provider data
  -> Activity canonical facts
  -> Sport Resolver
  -> sport-specific evidence resolver
  -> Record Registry match
  -> quality/confidence decision
  -> record comparison and state transition
  -> career_pb_records / career_record_events / candidates
  -> API ViewModel
  -> Records Center frontend / Snapshot / AI explanation
```

任何下游模块不得逆向修改上游事实。

## 2.2 Resolver 分层

V2 将纪录计算分为以下职责层：

| 层 | 职责 |
| --- | --- |
| Sport Resolver | 归一化 running/cycling/hiking/swimming/trail_running 等运动类型 |
| Activity Fact Adapter | 输出距离、时间、海拔、爬升、功率、Lap、轨迹可用性等白名单事实 |
| Activity-total Resolver | 生成最长距离、最大爬升、最高海拔等整次活动证据 |
| Power Curve Resolver | 生成骑行功率持续时间曲线和固定锚点证据 |
| Swim Effort Resolver | 从泳池 Length/Lap 或公开水域整次活动生成游泳证据 |
| Route/Segment Resolver | 生成路线签名、匹配同路线/赛段并计算 elapsed time |
| Record Resolver | 读取 Registry、比较新旧值、创建 candidate/active/superseded 事实 |
| Snapshot Builder | 只压缩正式纪录和安全摘要，不读取原始轨迹或功率流 |

## 2.3 写入边界

- 只有 Record Resolver/状态迁移服务可以写正式纪录。
- Curve、Route 和 Swim Resolver 只产出证据，不直接切换 active。
- 用户只能确认或拒绝候选，不能修改值、距离、时间、功率或来源。
- AI 不能创建、确认、拒绝、替换或删除纪录。
- 前端本地状态只能保存筛选、选中项和图表模式。

## 2.4 可追溯性

整次活动纪录至少包含：

```text
record_id -> activity_id -> record_key -> metric value -> evidence_key
```

分段、功率和路线纪录还必须包含：

```text
activity_id
start_index / end_index 或等价的 activity-relative range
duration_sec / distance_m
scope_key
resolver_version
quality reason codes
```

不得把 raw FIT、完整轨迹、文件路径或 SQLite schema 写入公开 ViewModel。

# 3. 术语、纪录族与生命周期

## 3.1 纪录族

| family | 含义 | 示例 |
| --- | --- | --- |
| `distance_time_pb` | 标准距离最快用时 | 跑步 10K、泳池 400m |
| `power_duration_pb` | 固定持续时间最高平均功率 | 骑行 5 分钟最佳功率 |
| `activity_total_record` | 整次活动最高/最长事实 | 最长徒步、最大爬升 |
| `route_pr` | 同路线最快 elapsed time | 越野路线 PR |
| `segment_pr` | 同赛段最快 elapsed time | 越野爬坡段 PR |
| `analysis_curve` | 只读能力曲线，不直接成为正式纪录 | 骑行 PDC、越野 GAP Curve |
| `model_estimate` | 模型估计，不是事实纪录 | eFTP、CP、NGP |

## 3.2 生命周期

继续使用：

```text
detected -> candidate -> active -> superseded
                  \-> rejected
active/superseded -> invalidated
```

新增曲线和路线缓存不具有 active 状态；只有从缓存/证据中选出的 Registry 锚点才进入正式纪录状态机。

## 3.3 比较原则

- `lower_is_better`：用时、配速型纪录。
- `higher_is_better`：功率、距离、爬升、海拔、时长型纪录。
- 相同值不创建新纪录，不替换 active。
- 不同 `scope_key` 永不互相比较。
- 模型估计与事实纪录永不互相替换。

# 4. Record Registry V2

## 4.1 Registry 定义

V2 的 `RecordDefinition` 至少包含：

```text
key
sport
family
display_name
metric
canonical_unit
comparison
source_mode
scope_dimensions
minimum_data_requirements
quality_policy
enabled_release
rule_version
priority
```

## 4.2 source_mode 扩展

| source_mode | 含义 |
| --- | --- |
| `activity_total` | 使用整次活动事实 |
| `best_effort_duration` | 固定持续时间的最佳平均值 |
| `best_effort_distance` | 固定距离的最佳连续用时 |
| `route_total` | 匹配同一路线后的整段用时 |
| `segment` | 活动内同一赛段的用时 |

## 4.3 scope_dimensions

V2 允许以下独立比较维度：

```text
sport_scope
indoor_scope
power_metric_scope
pool_length_scope
stroke_scope
water_scope
route_key
segment_key
```

所有 Scope 必须后端生成并进入 ViewModel；前端不得自行拼接 Scope。

## 4.4 Registry 约束

- 静态 `key` 发布后不可改变语义。
- 动态路线/赛段使用稳定 `record_key + scope_key`，不得把 route ID 拼成不可迁移的新类型。
- 新增单位、比较方向和 source_mode 必须通过 Registry 校验测试。
- 同一 Scope 中不得存在重复 active。
- 禁用定义只停止新评估，不删除历史。
- 未通过真实数据验收的定义在 Catalog 中必须标记 `available=false`，前端隐藏。

# 6. 骑行纪录规则

## 6.1 产品原则

骑行能力核心使用功率持续时间曲线，不使用固定距离速度作为唯一能力判断。距离、爬升和时长属于整次活动纪录；平均速度受风、路线、下坡、跟骑和活动长度影响，不作为 V2 核心正式纪录。

## 6.2 运动 Scope

默认功率曲线可聚合人力骑行：

```text
cycling / road_cycling / gravel / mountain_biking / indoor_cycling / virtual_ride
```

要求：

- 电助力骑行不与普通骑行混排。
- 前端允许全部、室内、户外筛选。
- 功率计/设备来源保留为过滤和质量证据，但不默认拆成多个公开纪录族。
- 没有逐点功率流的活动不能生成持续时间功率纪录。

## 6.3 正式功率锚点

底层可计算完整曲线，正式纪录只保存固定锚点：

| record_key | 持续时间 | 指标 | 比较 |
| --- | ---: | --- | --- |
| `cycling_power_5s` | 5s | 平均功率 W | 越高越好 |
| `cycling_power_30s` | 30s | 平均功率 W | 越高越好 |
| `cycling_power_1m` | 60s | 平均功率 W | 越高越好 |
| `cycling_power_5m` | 300s | 平均功率 W | 越高越好 |
| `cycling_power_10m` | 600s | 平均功率 W | 越高越好 |
| `cycling_power_20m` | 1,200s | 平均功率 W | 越高越好 |
| `cycling_power_30m` | 1,800s | 平均功率 W | 越高越好 |
| `cycling_power_60m` | 3,600s | 平均功率 W | 越高越好 |
| `cycling_power_2h` | 7,200s | 平均功率 W | 越高越好 |

1 秒功率对尖峰过于敏感，不进入 V2 正式锚点。

## 6.3.1 骑行标准距离最快成绩

| record_key | 显示名称 | 指标 | 比较 | 状态 |
| --- | --- | --- | --- | --- |
| `cycling_fastest_10k` | 最快 10K | elapsed_time_sec | 越低越好 | validation_required |
| `cycling_fastest_20k` | 最快 20K | elapsed_time_sec | 越低越好 | validation_required |
| `cycling_fastest_40k` | 最快 40K | elapsed_time_sec | 越低越好 | validation_required |
| `cycling_fastest_50k` | 最快 50K | elapsed_time_sec | 越低越好 | validation_required |
| `cycling_fastest_100k` | 最快 100K | elapsed_time_sec | 越低越好 | validation_required |
| `cycling_fastest_180k` | 最快 180K | elapsed_time_sec | 越低越好 | validation_required |

标准距离必须来自 Activity 内距离-时间流的固定距离窗口；不得用整次活动均速、整次活动 elapsed time 或前端计算结果替代。距离-时间流、暂停语义、室内外里程来源和异常 GPS/速度跳点规则未冻结前，只能作为 Catalog 可见的待验证入口，不生成 active record。

## 6.4 功率曲线算法契约

```text
best_power(duration) = max(time_weighted_average(power, contiguous_window))
```

- 使用真实时间戳和时间加权平均，不假定所有采样严格 1Hz。
- 0W 是有效滑行数据；缺失值不是 0W。
- 窗口不得跨越长暂停、数据断裂或无效时间戳。
- 尖峰检测必须在纪录比较前执行。
- 每个锚点保存来源 Activity 和起止索引。
- 汇总 `avg_power/max_power/normalized_power` 不能替代逐点曲线。
- 采样间隔、断点和尖峰阈值必须配置化并由真实数据审计冻结。

## 6.5 W/kg

- W/kg 是与绝对 W 分开的 `power_metric_scope`。
- 只有活动日期附近存在可靠历史体重时才生成。
- 当前体重不得回填历史活动。
- 体重来源、日期和质量等级必须进入证据，但不得在公开 ViewModel 暴露隐私细节。
- 无可靠体重时只显示 W，不创建 W/kg 候选。

## 6.6 骑行整次活动纪录

| record_key | 显示名称 | 指标 | 比较 |
| --- | --- | --- | --- |
| `cycling_longest_distance` | 最长骑行距离 | distance_m | 越高越好 |
| `cycling_max_ascent` | 单次最大爬升 | ascent_m | 越高越好 |
| `cycling_longest_elapsed_time` | 最长骑行历时 | elapsed_time_sec | 越高越好 |
| `cycling_max_work` | 单次最大机械功 | work_kj | 越高越好 |

机械功只有在功率流或可信设备汇总可用时生成。最快均速推迟到有最低距离、室内外和路线公平性规则后再评估。

## 6.7 模型边界

eFTP、CP、W′、PMax、MAP 和排名属于模型/分析输出：

- 不写入 active PB。
- 不替代功率锚点。
- 可以在详情中标注“模型估计”。
- AI 不得把模型变化写成“刷新纪录”。

# 7. 徒步纪录规则

## 7.1 运动边界

`hiking`、`walking` 和 `mountaineering` 分开。V2 徒步纪录只接收归一化为 `hiking` 的 Activity；城市步行、室内步行、登山和越野跑不进入徒步纪录。

## 7.2 正式纪录

| record_key | 显示名称 | 指标 | 比较 |
| --- | --- | --- | --- |
| `hiking_longest_distance` | 最长徒步距离 | distance_m | 越高越好 |
| `hiking_max_ascent` | 单次最大累计爬升 | ascent_m | 越高越好 |
| `hiking_longest_elapsed_time` | 最长徒步历时 | elapsed_time_sec | 越高越好 |
| `hiking_max_altitude` | 最高到达海拔 | altitude_m | 越高越好 |
| `hiking_max_single_climb` | 最大连续爬升 | climb_gain_m | 越高越好 |

“最长徒步历时”冻结为 elapsed time，真实反映从开始到结束的户外经历；moving/timer time 仅作为详情上下文。

## 7.3 不设置通用最快配速

- 不设置跨路线最快 5K/10K。
- 不用平均配速评价不同坡度、路况和负重的活动。
- 同路线最快用时必须等待 Route Resolver，并使用独立 route Scope。
- 爬升速度/VAM 可作为分析指标，V2 不作为通用正式纪录。

## 7.4 数据质量

- 距离必须大于 0 且单位明确。
- 海拔和爬升需优先使用设备/规范化事实；原始 GPS 高度尖峰不能自动确认。
- 最大连续爬升必须由连续海拔曲线生成，不能用整次累计爬升冒充。
- 轨迹缺失时可参与距离/历时，不能参与最大连续爬升。
- 旧数据时间口径未知时进入候选。

# 8. 游泳纪录规则

## 8.1 Scope 分离

```text
pool_swimming
open_water_swimming
```

两个 Scope 永不互相比较，也不共享 active。

## 8.2 泳池游泳标准距离

V2 首先支持明确为自由泳的米制泳池活动：

| record_key | 距离 | 指标 | 比较 |
| --- | ---: | --- | --- |
| `pool_swim_50m` | 50m | segment elapsed time | 越低越好 |
| `pool_swim_100m` | 100m | segment elapsed time | 越低越好 |
| `pool_swim_200m` | 200m | segment elapsed time | 越低越好 |
| `pool_swim_400m` | 400m | segment elapsed time | 越低越好 |
| `pool_swim_800m` | 800m | segment elapsed time | 越低越好 |
| `pool_swim_1500m` | 1,500m | segment elapsed time | 越低越好 |

## 8.3 泳池 Scope

- 25m 池和 50m 池分别使用 `scm_25m`、`lcm_50m`。
- 码制泳池不进入 V2 首次发布。
- 距离由 `valid_length_count × pool_length_m` 生成，不使用 `±3%`。
- 缺少 pool length 时不得默认 25m 生成正式纪录。
- 自由泳以外泳姿需要独立 `stroke_scope`；V2 首发只自动确认显式自由泳。
- 混合泳姿或泳姿未知进入候选，不与自由泳混排。

## 8.4 泳池分段算法

- 使用连续有效 Length/Lap 组合固定距离。
- 组间休息会中断最佳努力窗口。
- 计时包含转身，不包含训练组之间的静止休息。
- 每个结果保存 Activity、起止 Length/Lap 或等价索引。
- 只有活动汇总距离和总时长、没有 Length/Lap 证据时不能生成泳池 PB。

## 8.5 公开水域

V2 标准距离：

```text
750m / 1500m / 1900m / 3800m / 5K / 10K
```

V2.0 使用整次活动匹配，允许包含边界的 `±5%` 距离误差；该阈值必须在真实样本审计后复核。正式比较使用 elapsed time，越低越好。

同时支持：

| record_key | 显示名称 | 指标 | 比较 |
| --- | --- | --- | --- |
| `open_water_longest_distance` | 最长公开水域距离 | distance_m | 越高越好 |
| `open_water_longest_elapsed_time` | 最长公开水域历时 | elapsed_time_sec | 越高越好 |

要求：

- 必须明确为公开水域。
- 标准距离自动确认需有可靠 GPS、连续时间和无明显跳点。
- 手动距离、轨迹缺失或距离来源不明最多进入候选。
- 水流、浪、风和水温只作上下文，不改变正式用时。

## 8.6 SWOLF

SWOLF 不作为 V2 首发正式纪录。它可作为详情分析指标，但不同泳池长度、泳姿和距离不得比较。若未来进入记录中心，必须绑定 pool length、stroke 和距离 Scope。

# 9. 越野跑纪录规则

## 9.1 运动边界

```text
road_running != trail_running != hiking != mountaineering
```

- 只有 Sport Resolver 明确归一为 `trail_running` 才参与。
- 标题出现“越野”不能单独构成高置信度事实。
- 类型不明确但证据较强时进入候选。
- 越野活动不进入公路跑四项 PB。

## 9.2 整次活动纪录

| record_key | 显示名称 | 指标 | 比较 |
| --- | --- | --- | --- |
| `trail_longest_distance` | 最长越野距离 | distance_m | 越高越好 |
| `trail_max_ascent` | 单次最大累计爬升 | ascent_m | 越高越好 |
| `trail_longest_elapsed_time` | 最长越野历时 | elapsed_time_sec | 越高越好 |
| `trail_max_altitude` | 最高到达海拔 | altitude_m | 越高越好 |
| `trail_max_single_climb` | 最大连续爬升 | climb_gain_m | 越高越好 |

## 9.3 路线与赛段 PR

| family | 唯一 Scope | 比较指标 |
| --- | --- | --- |
| `trail_route_best_time` | `route_key + direction` | elapsed time 越低越好 |
| `trail_segment_best_time` | `segment_key + direction` | elapsed time 越低越好 |
| `trail_climb_segment_best_time` | `segment_key + direction` | elapsed time 越低越好 |

每条路线/赛段 PR 必须绑定 Activity 和起止索引。V2 不建立公开排行榜，只比较用户自己的历史。

## 9.4 路线匹配初始策略

以下值作为待真实样本审计的配置默认值，不得散落硬编码：

- 起点和终点分别在 100m 容差内。
- 路线长度差异不超过 5%。
- 轨迹有效覆盖率至少 95%。
- 同方向轨迹走廊重合度至少 85%。
- 反向、折返差异、GPS 长缺口或明显跳点不自动确认。

在没有真实越野样本通过审计前，Route Resolver 可以开发和 dry-run，但所有路线匹配结果只能候选。

## 9.5 Pace/GAP Curve

参考 Intervals.icu，可提供原始 Pace Curve 与 GAP Curve，按全部时间、当前赛季、最近 42 天比较。

```text
1K / 3K / 5K / 10K / 20K / 30K / 50K
```

曲线属于分析视图，不直接写入正式 PB：

- GAP/NGP 不考虑泥泞、碎石、台阶、积雪和技术难度。
- 不同海拔、气候和负重不能靠单一坡度模型完全校正。
- 页面可以写“GAP 曲线变化”，不得写“刷新越野 10K 正式纪录”。

# 10. 置信度与原因码

## 10.1 通用区间

| confidence | 处理 |
| ---: | --- |
| `>0.90` | 自动确认并进入正式比较 |
| `0.70-0.90` | 候选，不替换 active |
| `<0.70` | 忽略或仅留审计原因 |

## 10.2 评分维度

- 运动类型可靠性。
- 距离、时间和单位可靠性。
- 轨迹/功率/Lap 采样完整度。
- 设备或传感器来源。
- Scope 完整度。
- 尖峰、断点、暂停和异常值。
- 旧字段语义是否明确。
- Resolver 版本和证据可复算性。

## 10.3 V2 原因码建议

```text
missing_power_stream
power_spike_detected
power_stream_gap
historical_weight_missing
ebike_scope_excluded
elevation_source_unknown
elevation_spike_detected
timer_semantics_unknown
pool_length_missing
unsupported_pool_unit
swim_stroke_unknown
swim_laps_missing
open_water_gps_unreliable
trail_sport_ambiguous
route_match_low_overlap
route_direction_mismatch
track_coverage_insufficient
```

原因码由后端翻译为安全 ViewModel 文案，前端不解析技术 payload。

# 11. 数据模型与缓存

## 11.1 兼容策略

- `career_pb_records` 继续作为正式纪录表。
- `pb_type` 保持兼容，可在服务层映射为 `record_key`。
- 新增结构化 Scope 和分段字段，不把关键语义只塞入 JSON。
- `career_record_events` 和候选表沿用现有状态语义。

## 11.2 建议新增字段

```text
record_family
scope_key
metric_direction
duration_sec
distance_m
start_index
end_index
route_key
segment_key
pool_length_m
stroke_scope
indoor_scope
quality_level
quality_reason_codes_json
```

字段命名需在 schema audit 后冻结；已有字段能表达相同语义时优先复用。

## 11.3 Curve Cache

允许新增不与正式纪录表重叠的派生缓存，例如：

```text
career_record_curve_cache
  activity_id
  curve_type
  scope_key
  algorithm_version
  source_fingerprint
  durations_or_distances
  values
  source_ranges
  created_at
```

要求：

- 缓存不是 canonical record。
- Activity 修改/删除或算法版本变化时失效。
- 缓存不得复制 raw FIT、完整轨迹或本地路径。
- 前端只能通过安全 Curve ViewModel 读取。

## 11.4 Route 数据

路线匹配可以保存派生签名和匹配关系，但不得创建公开社交排行榜。原始轨迹继续属于 Activity，不在记录中心事实表重复保存。

## 11.5 唯一性

active 唯一键应覆盖：

```text
record_key + sport_scope + scope_key + source_mode
```

相同 Activity、相同证据、相同 Resolver 版本必须幂等。

# 12. Resolver 执行流程

## 12.1 增量导入

```text
Activity imported/updated
  -> normalize sport and facts
  -> choose applicable V2 evidence resolvers
  -> build/cache curve or route evidence when required
  -> match Record Registry
  -> score quality/confidence
  -> candidate / ignored / formal comparison
  -> transactional state transition
  -> append record event
  -> refresh safe snapshot and UI notification
```

## 12.2 删除和修正

- 删除 Activity 会失效其整次、曲线锚点、泳段、路线和赛段纪录。
- 删除 active 后从相同 Scope 剩余有效记录中选下一条 active。
- Activity 修正先使旧证据失效，再以新 fingerprint 重新评估。
- Curve/Route 缓存同步失效。

## 12.3 全量重建

- 必须支持 plan/dry-run/apply。
- dry-run 返回新增、替换、候选、忽略、失效和 Scope 变化摘要。
- apply 使用单一受控写入口和事务。
- 重建期间旧结果保持可读。
- 未经用户确认不得对真实库执行 V2 apply。

# 13. API 与 ViewModel

## 13.1 兼容 API

以下 API 保持存在：

```text
get_career_pb
get_career_pb_detail
get_career_pb_history
get_career_event_candidates
decide_career_pb_candidate
```

V1 调用方的字段语义不得被 V2 改坏。

## 13.2 V2 通用只读 API 建议

```text
get_career_record_catalog()
get_career_records(filters)
get_career_record_detail(record_id)
get_career_record_history(record_key, scope_key, filters)
get_career_record_curve(payload)
```

旧 PB API 可以作为兼容包装器，新前端优先消费通用 Records ViewModel。是否新增 API 需先更新 `docs/js_api_contract.json`。

## 13.3 Catalog ViewModel

Catalog 决定哪些运动和纪录可见：

```json
{
  "sports": [
    {
      "sport": "cycling",
      "label": "骑行",
      "available": true,
      "families": ["power_duration_pb", "activity_total_record"],
      "record_keys": ["cycling_power_5s", "cycling_power_30s"]
    }
  ]
}
```

未完成或未通过数据验收的运动必须 `available=false`，前端不得仅凭硬编码展示。

## 13.4 Record ViewModel

```text
id
activity_id
record_key / record_family
sport / sport_label
title
value / unit / value_display
improvement / improvement_display
event_date / display_date
status / status_label
scope_key / scope_label
source_mode / source_mode_label
confidence / confidence_label
quality_label / evidence_summary
detail_link {activity_id, source, record_id}
```

分段详情可以额外返回安全的 activity-relative range，但不得返回 raw points。

## 13.5 History Summary

History API 应由后端提供：

```text
first_record
current_record
total_improvement_display
refresh_count
valid_record_count
```

前端不自行计算累计提升。

## 13.6 Curve ViewModel

Curve API 返回派生曲线、时间范围、来源 Activity 引用和可用性状态；不返回 raw 功率流、GPS 轨迹或文件路径。

# 14. 前端视觉与交互

## 14.1 视觉方向

V2 采用已评审的深色运动数据看板作为视觉参考：左侧纪录选择，右侧大图表和摘要，清晰的选中态、低饱和边框和单一强调色。

只参考视觉结构，不复制其模拟数据、表现指数、外部 CDN、独立应用顶栏或错误运动语义。

## 14.2 页面信息架构

```text
运动：跑步 ｜ 骑行 ｜ 越野跑 ｜ 徒步 ｜ 游泳

视图：当前纪录 ｜ 演进 ｜ 候选

桌面：左侧分组纪录列表 + 右侧主图/详情
窄窗口：横向纪录选择器 + 单列主图/详情
```

运动页签来自 Catalog，未完成运动不显示。

## 14.3 左侧纪录列表

- 展示真实值、日期、提升量和状态，不展示“基准 100”表现指数。
- 数量较多时按纪录族分组并允许滚动/折叠。
- 选中态不能只依赖颜色。
- 卡片点击选择纪录；独立主操作进入 Activity Detail。

## 14.4 右侧图表模式

| 运动 | 主图模式 |
| --- | --- |
| 跑步 | 纪录演进阶梯图 |
| 骑行 | 纪录演进 / 功率曲线；W/Wkg；全部/赛季/42天；室内外 |
| 徒步 | 距离、爬升、海拔、历时的纪录演进 |
| 游泳 | 泳池/公开水域；标准距离纪录演进 |
| 越野跑 | 纪录演进 / 路线对比 / Pace-GAP Curve |

跑步时间越低越好，图上必须视觉向上；功率、距离和爬升越高越好。轴方向由后端 Registry/ViewModel 决定。

## 14.5 图表要求

- 纪录历史使用真实事件节点，不按年份补造数据。
- 跑步/游泳用时图优先使用阶梯线或明确反向轴。
- 功率曲线横轴使用持续时间并允许对数尺度。
- Tooltip 显示格式化值、日期、提升、Scope 和来源活动入口。
- invalidated 使用空心/虚线状态，不从历史中静默消失。
- 只有一条纪录时显示“首次建立纪录”，不伪造趋势。
- 图表必须有可访问的历史节点列表替代视图。

## 14.6 时间和过滤控件

参考设计中的“年度｜季度｜月度”不直接照搬。按运动使用：

```text
全部时间 ｜ 当前赛季 ｜ 最近42天 ｜ 自定义范围
W ｜ W/kg
全部骑行 ｜ 室内 ｜ 户外
泳池 ｜ 公开水域
原始配速 ｜ GAP
```

## 14.7 候选视图

候选至少展示：纪录类型、可能值、Scope、来源 Activity、未自动确认原因、置信度和确认/拒绝操作。模型估计、分析曲线和不具备正式纪录资格的数据不得混入候选。

## 14.8 页面状态

所有运动必须覆盖：

```text
Loading / Empty / Partial / Candidate / Rebuilding / Error / Validation Required
```

没有功率计、没有泳池样本或没有越野真实数据时使用说明性空态，不显示虚构卡片。

## 14.9 离线与响应式

- 使用应用本地 ECharts、字体和图标。
- 不依赖墨刀、Google Fonts、远程头像或运行时 Tailwind CDN。
- 1440px、1100px、980px、720px 和窄窗口均需截图验收。
- 支持键盘切换运动、视图和纪录；按钮有名称、状态和 tooltip。
- 遵守 `prefers-reduced-motion`。

# 15. Overview、Timeline、Achievement、Trends 与 AI

## 15.1 Overview

- 只统计正式 active/superseded 纪录。
- 候选不进入 PB 数量、最新纪录或代表纪录。
- 多运动摘要应显示 sport/family，不把最长距离和标准距离 PB 混成同一类型。

## 15.2 Timeline

- 只有正式纪录首次建立或刷新才产生 Timeline/通知候选事件。
- 曲线缓存变化、模型变化、候选、拒绝和 dry-run 不进入主 Timeline。
- 路线/赛段 PR 必须通过产品门禁后才进入 Timeline。

## 15.3 Achievement

正式纪录可触发 Achievement，但同一 record event 必须幂等。模型估计和分析曲线不触发 Achievement。

## 15.4 Trends

Trends 可以消费功率曲线、Pace/GAP Curve 和刷新频率，但不得反向写正式纪录。趋势结论必须标注事实曲线或模型来源。

## 15.5 AI

AI 只消费安全 Records Snapshot：

- 当前正式纪录。
- 最近刷新。
- 候选数量，不含敏感证据详情。
- 纪录演进摘要。
- 曲线变化摘要和可用性。

禁止：

- 读取 raw FIT、功率流、轨迹、文件路径或 schema。
- 把 GAP/NGP/eFTP 等模型写成正式纪录。
- 把候选说成已确认。
- 根据没有真实样本的运动生成强结论。

# 16. 性能、安全与可观测性

## 16.1 性能

- 功率、Pace、泳段和路线证据按 Activity fingerprint 增量缓存。
- 列表查询不得扫描 raw track/power points。
- Curve 查询优先读缓存。
- 全量重建必须可分批、可取消、可汇总。
- 记录中心加载失败不得阻塞 Activity 主功能。

建议目标：

| 路径 | 目标 |
| --- | --- |
| 当前纪录列表 | P95 < 150ms |
| 单项历史 | P95 < 200ms |
| Curve 缓存命中 | P95 < 250ms |
| 候选列表 | P95 < 150ms |

## 16.2 安全

- API 不返回 raw points、track_json、FIT path、SQLite schema、传感器序列号或体重历史详情。
- 路线签名不得泄露完整用户轨迹。
- 高风险 apply/confirm/reject 接口继续要求输入校验、确认和幂等。
- 日志不输出完整 evidence payload 或本地路径。

## 16.3 可观测性

至少记录：

```text
processed / active / superseded / candidate / ignored / invalidated
by_sport / by_family / by_reason_code
curve_cache_hit / curve_cache_miss
route_match_candidate_count
resolver_version / elapsed_ms
```

# 17. 迁移与真实数据发布策略

## 17.1 Schema migration

- 先做 schema/索引 dry-run 和回滚测试。
- migration 可重复执行。
- V1 现有跑步 active/history 不改变。
- 新 Scope 唯一索引必须先审计历史冲突。

## 17.2 Backfill 顺序

```text
Catalog/Registry
  -> activity-total records
  -> cycling power curves
  -> pool/open-water evidence
  -> trail route/segment evidence
  -> comparison plan
```

## 17.3 真实库门禁

1. 备份真实库并校验哈希。
2. 在 staging 副本运行 V2 dry-run。
3. 输出按运动、纪录族、原因码分类的计划。
4. 人工抽查来源 Activity 和异常样本。
5. 第一次 V2 真实数据结果默认保持候选/计划态。
6. 只有用户明确批准后才允许 apply。
7. apply 后验证 active 唯一性、历史链和删除回退。

## 17.4 样本不足

- 无泳池样本：只能通过 fixture，不能标记真实数据 Verified。
- 无越野样本：Route/Segment 只能 candidate-only。
- 某运动未通过真实验收时，Catalog 必须隐藏或标记预览，不得伪装已完成。

# 18. 测试策略

## 18.1 Registry 单元测试

- key/Scope 唯一性。

# 19. 验收标准

## 19.1 产品验收

- 记录中心视觉明显区别于旧紧凑列表，但仍属于脉图现有应用外壳。
- 用户能按运动和纪录族理解“当前值、来源、演进和候选”。
- 不显示未交付运动或模拟数据。
- 模型估计、分析曲线和正式纪录有清晰区别。

## 19.2 规则验收

- 跑步 V1 结果不变。
- 骑行锚点来自逐点功率窗口。
- 徒步不与步行/登山混排。
- 泳池/公开水域、25m/50m、泳姿 Scope 不混排。
- 越野正式 PR 仅在同路线/同赛段内比较。
- GAP/NGP/eFTP 等不进入正式 active。

## 19.3 数据验收

- 每条纪录可回溯 Activity；分段纪录可回溯范围。
- active 唯一、历史完整、删除可回退。
- 真实库只在用户批准后 apply。
- 样本不足运动不会被标记为 Verified。

## 19.4 工程验收

- API contract、schema、Registry、UI 和测试一致。
- 目标测试和宽回归全绿。
- 无敏感数据泄露。
- Curve/Route 性能符合目标。
- macOS 和 Windows 重新验收完成后才能发布。

# 21. Definition of Done

V2 只有在以下条件全部满足时才可标记完成：

1. V1 跑步纪录无回归。
2. V2 Registry、Scope、单位、方向和版本均已冻结并测试。
3. 骑行、徒步、游泳、越野跑按本文规则实现或在 Catalog 中明确不可用。
4. 当前、演进、候选三条路径可用。
5. 分析曲线和模型估计未污染正式纪录。
6. 所有正式纪录可追溯到 Activity，分段纪录可追溯范围。
7. 候选和异常原因可解释、可确认、可拒绝且幂等。
8. 真实数据只在用户授权后写入。
9. 前端无模拟表现指数、无未交付占位、无外部运行时依赖。
10. 定向测试、宽回归、视觉、macOS 和 Windows 门禁均有证据。
11. API contract、手册、任务清单、完成报告和发布说明一致。
12. 用户暂停打包的决定被明确解除后，才可生成新的发布包。
