# Part I：系统概述

## 1.1 这是什么

运动复盘系统（Activity Review & Fatigue Analysis）是对 FIT 格式运动数据（跑步/骑行/徒步/越野）进行**生理状态演化分析**的模块。

不是普通的数据图表，而是回答：

```
身体状态如何变化？
↓
为什么失衡？
↓
在哪里开始崩？
↓
什么因素导致崩溃？
```

## 1.2 核心输出

| 输出 | 说明 |
|------|------|
| 疲劳带（Fatigue Zones） | Layer 1，风险区段可视化 |
| 事件标记（Event Markers） | Layer 2，关键生理事件点 |
| 派生指标曲线（Derived Metrics） | Layer 3，Efficiency / HR Drift / Decoupling 等 |
| 洞察文本（AI Insight） | LLM 生成的自然语言解释 |

## 1.3 数据流

```
FIT Raw Signals
    ↓
Preprocessing Layer（resample / smooth / gradient / GAP）
    ↓
Derived Metrics Layer（Efficiency / HR Drift / Decoupling / ...）
    ↓
State Engine（疲劳状态 5 级判定）
    ↓
Event Detection（Drift Start / Efficiency Drop / Bonk / ...）
    ↓
Fatigue Score（加权综合评分）
    ↓
Insight Engine（规则 + LLM）
    ↓
UI Visualization（ECharts 多图层叠加）
```

## 1.4 项目文件结构

| 文件 | 职责 |
|------|------|
| `main.py` | Flask 路由 + `_build_fatigue_review_snapshot()` |
| `metrics_resolver.py` | 核心解析引擎（Resolver） |
| `gap_calculator.py` | GAP 等效配速计算 |
| `utils/metrics_calc.py` | 高级指标计算（Decoupling / VAM / Threshold HR） |
| `llm_backend.py` | LLM 调用 + prompt 构建 |
| `track.html` | 前端 UI（ECharts 多图层） |

# Part II：架构契约（强制遵守）

> ⚠️ **这些契约是硬性约束，不是建议。** 违反任何一条，代码将被拒绝合并。

## §2.1 FIT 为唯一事实源

所有活动数据最终必须能够回溯到：
- 原始 FIT 文件
- Garmin FIT SDK 字段定义
- `fit_engine.py` 解析输出

**禁止**：
- 前端生成虚假指标
- AI 生成伪运动数据
- UI fallback 长期污染数据层
- synthetic 数据进入 canonical layer

## §4.5 AI Snapshot Layer

**目标**：将复杂运动数据压缩为 AI 可消费上下文。

**规则**：
- AI **不应该**直接读取原始 FIT、原始 records、SQLite schema
- AI **只应该**读取 snapshot
- Snapshot 必须：小、稳定、可解释、token 可控

## §5 字段全链路可追溯

**规则**：UI 字段必须能追溯至 FIT SDK 原始字段。

示例：`metrics.efficiency.score` → `Efficiency = Speed / HR` → FIT `record.heart_rate` + `record.speed`

**禁止**：前端偷偷计算、AI 偷偷生成、Resolver 无来源推断。

## §6 Shadow Diff 隔离

**规则**：`shadow_diff` / `shadow_diff_json` / `diff` 字段**严禁**进入前端 JSON 契约。

后端 `resolve()` 出口必须有白名单过滤：
```python
for forbidden in ("shadow_diff", "shadow_diff_json", "diff"):
    resolved.pop(forbidden, None)
```

## §8.1 禁止架构膨胀

**明确不引入**：
- Data Warehouse
- Kafka / Event Bus
- Feature Store
- Runtime Orchestrator
- 微服务体系
- 云端计算节点
- SaaS 多租户架构

**原因**：脉图是**单用户、本地 AI 工具**。

## §9.4 前端修改约束

**规则**：
- 只改 CSS，不改 HTML 结构
- 所有提示词必须内置「零、架构契约核对」章节
- `INSERT` 语句列数 = 占位符数（用临时 print debug 验证）

### 4.1.3 Gradient（坡度）

```
Gradient = ΔAltitude / ΔDistance
```

输出：`grade_percent[]`

状态：✅ 已实现（`gap_calculator.py::_calculate_grade_pct()`）

### 4.1.4 GAP（Grade Adjusted Pace）

基于 **Minetti 能量消耗多项式**的非线性计算。

```
等效配速 = f(真实配速, 坡度, 运动类型)
```

- 越野/徒步：使用 GAP
- 路跑/骑行：不使用 GAP

输出：`gap_curve[]`

状态：✅ 已实现（`gap_calculator.py::_calculate_gap_speed_non_linear()`）

## 5.1 Efficiency（效率指数）⭐ 核心

### 生理意义

单位心率代价下的输出能力。值越高 = 同样心率能输出更高速度 = 效率越好。

### 公式

| 运动类型 | 公式 |
|---------|------|
| 跑步 | `Speed / HR` |
| 骑行 | `Power / HR` |
| 越野 | `GAP Speed / HR` |

### 算法步骤

1. 逐点计算 `ei = speed / hr`（过滤无效值）
2. 取前半程基线 EI
3. 扫描后半程，检测 EI 断崖下跌
4. 与 21d baseline 比较，输出 `delta_pct`

### 输出 JSON 结构

```json
{
  "metrics": {
    "efficiency": {
      "score": 52.3,
      "level": "improving|stable|declining|unknown",
      "confidence": "high|medium|low|unavailable",
      "delta_pct": 3.5,
      "sample_size": 8,
      "trend": {
        "level": "flat|up|down",
        "compared_count": 5,
        "baseline_ratio": 0.0042,
        "source": "v7_14_baseline"
      }
    }
  }
}
```

### 阈值

| delta_pct | 等级 |
|-----------|------|
| > 5% | improving |
| ±5% | stable |
| < -5% | declining |

### 后端函数（代码实际）

- 基础比值：`MetricsResolver._compute_efficiency_score()` （`metrics_resolver.py` L1850）
- baseline 比对：`evaluate_efficiency()` （module-level 公开 API，`metrics_resolver.py` L2517）
- baseline 来源：`_fetch_efficiency_baseline()` → 21d 中位数

### UI 表达

- Layer 3 Derived Metrics：绿色效率曲线
- KPI 卡片：数值 + 评级（improving/stable/declining）
- 颜色：绿色（improving）、灰色（stable）、橙色（declining）

## 5.2 HR Drift（心率漂移）⭐ 核心

### 生理意义

相同输出下，HR 持续升高 → 心血管系统效率下降（热应激/脱水/疲劳）。

### 稳态窗口识别（设计文档要求）

| 窗口 | 规则 |
|------|------|
| Early Window | 去掉前 10%，在前 40% 中找 HR variance 最低 + pace variance 最低，取连续 5 分钟 |
| Late Window | 去掉最后 5%，在后 40% 中找 HR variance 最低 + pace variance 最低，取连续 5 分钟 |

### 公式

```
HR_Drift_% = (HR_late - HR_early) / HR_early × 100
```

### ⚠️ 代码实际实现（与设计文档有差异）

| 项目 | 设计文档 | 代码实际 |
|------|---------|---------|
| 窗口 | Early/Late Window（稳态识别） | 前后半程简单切分 |
| 阈值 | <3% Stable / 3-5% Mild / 5-8% Drift / >8% Severe | <5% excellent / 5-10% good / 10-15% warn / >15% bad |

**函数**：`MetricsResolver._compute_hr_drift()` （`metrics_resolver.py` L2401）

**前置检查**：`_classify_steady_aerobic()` — duration > 45min + 配速 CV < 8% + 停顿 < 10%

### 输出 JSON 结构

```json
{
  "metrics": {
    "hr_drift": {
      "pct": 4.5,
      "level": "excellent|good|warn|bad|unknown",
      "confidence": "high|medium|low|unavailable",
      "trend": {
        "delta_pct": -2.1,
        "level": "up|down|flat",
        "compared_count": 5,
        "is_improving": true,
        "source": "historical_avg"
      }
    }
  }
}
```

## 5.3 Aerobic Decoupling（有氧解耦率）⭐ 核心

### 生理意义

输出与生理代价开始失配。正常情况下，相同配速下 HR 应稳定；若 HR 升高或配速下降，说明效率降低。

### 🔴 公式方向争议

| 维度 | 设计文档 | 代码实际 |
|------|---------|---------|
| ratio 定义 | `Speed / HR`（输出/代价） | `HR / Speed`（代价/输出） |
| 语义 | "效率"方向：值越高越好 | "代价"方向：值越高越差 |
| decoupling 公式 | `(S/HR_early)/(S/HR_late) - 1` | `(r2-r1)/r1`，其中 `r = HR/Speed` |

**分析**：两种方向在数学上等价（后半程效率下降时都返回正值），但语义解释不同。

**函数**：`AdvancedMetricsCalc.calculate_aerobic_decoupling()` （`utils/metrics_calc.py` L83）

### 阈值

| Decoupling | 状态（代码实际） |
|-----------|------------------|
| < 5% | excellent |
| 5-10% | good |
| 10-15% | warn |
| > 15% | bad |

### 输出 JSON 结构

```json
  "metrics": {
    "decoupling": {
      "pct": 3.2,
      "level": "excellent|good|warn|bad|unknown",
```

## 5.5 TRIMP / Training Load（训练负荷）⭐ 核心

### 生理意义

内部生理负荷量化。

### 公式

**设计文档**（经典 TRIMP）：
```
TRIMP = Duration × HR_ratio × e^(k × HR_ratio)
HR_ratio = (HR - HR_rest) / (HR_max - HR_rest)
```

**代码实际**（Banister 简化版）：
```
load = duration_min × Σ(zone_weight × time_pct / 100)
无 zone 分布时降级：avg_hr/max_hr → zone 映射
```

### 函数

`MetricsResolver._compute_training_load()` （`metrics_resolver.py` L2069）

### 输出 JSON 结构
