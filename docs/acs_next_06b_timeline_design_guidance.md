# ACS-Next-06B Timeline 核心浏览体验设计指导

本文用于沉淀 ACS 时间轴页面的产品设计、事件口径和工程边界，作为后续系列任务拆解、实现和验收的参考依据。

若本文与以下契约冲突，以项目契约原文和 API 契约为准：

1. `docs/脉图运动生涯系统（ACS）开发团队交付手册.md`
2. `docs/脉图运动生涯系统（ACS）开发任务清单.md`
3. `docs/js_api_contract.json`
4. `.trae/rules/fit-arch-contrac.md`

## 1. 核心契约

- `Activity` 是唯一事实源。
- ACS 只组织派生展示和安全媒体引用，不复制 raw FIT 事实。
- 前端不得自行推断赛事、PB、成就、里程碑或训练事实。
- 不得向前端、AI、Snapshot 或 API 返回 raw FIT、points、track_json、file_path、storage_ref、本地绝对路径、SQLite schema。
- Timeline 只展示后端 Resolver / Timeline ViewModel 已确认的正式节点。
- 低置信度候选事件不进入正式时间轴。
- 每个可点击节点必须能回跳到 Activity Detail，除非该节点是明确的全局累计节点且后端提供了达成节点对应的 `activity_id`。
- Windows 真机、Windows 打包、macOS 打包产物验证未执行前，不得标记为打包级完成。

## 2. 产品目标

时间轴页面不再是普通节点列表，也不再展示年度卡片或“高光年”。它应成为用户按时间回看运动生涯的核心浏览方式。

目标体验：

- 顶部用年份胶囊快速切换时间范围。
- 顶部用内容胶囊切换 `全部 / 赛事 / 里程碑`。
- 主体按月份组织。
- 每个月内部使用横向日期轴，从左到右表示当月 1 日到 31 日。
- 同一个月份内，不同内容类型占用不同轨道：
  - 第一行：赛事
  - 第二行：里程碑
- 事件卡片根据发生日期落在对应横向位置。
- 页面宽度被充分利用，用户能直观看到“这个月哪些日子发生了什么”。

## 3. 顶部筛选设计

### 3.1 年份筛选

年份筛选使用胶囊按钮：

- 默认：`全部`
- 其后按倒序展示已有年份，例如 `2026 / 2025 / 2024`
- 年份来源必须来自后端 Timeline ViewModel 或安全聚合字段。
- 前端可以渲染和传递筛选参数，但不得自行扫描原始活动推断年份。

交互：

- 点击 `全部` 展示所有年份下的所有月份。
- 点击某一年只展示该年份的月份。
- 切换年份时保留当前内容筛选类型。

### 3.2 内容筛选

内容筛选使用胶囊按钮：

- `全部`
- `赛事`
- `里程碑`

交互：

- `全部`：展示赛事轨道和里程碑轨道。
- `赛事`：只展示赛事轨道。
- `里程碑`：只展示里程碑轨道。

前端只传递筛选类型，不在本地重新计算节点归属。

## 4. 月份横向时间轴布局

### 4.1 月份容器

每个月是一个独立横向时间带：

- 左侧固定显示月份，例如 `07 JUL`、`06 JUN`。
- 右侧为日期轴区域。
- 日期轴按当月天数计算，不固定假设 31 天。

横向定位建议：

```text
left_percent = (day - 1) / max(days_in_month - 1, 1) * 100
```

说明：

- 1 日落在最左侧。
- 当月最后一天落在最右侧。
- 15 日左右落在中间区域。
- 闰年 2 月必须由后端或前端安全日期工具按实际天数计算。

### 4.2 轨道

同一个月份内建议使用固定轨道：

```text
赛事轨道
里程碑轨道
```

当筛选为 `赛事` 时，只显示赛事轨道。

当筛选为 `里程碑` 时，只显示里程碑轨道。

当某条轨道无事件时，可以隐藏该轨道或保留轻量空态。默认建议隐藏空轨道，避免页面显得稀疏。

### 4.3 卡片避让

同一天或日期接近的卡片可能重叠。后续实现需要至少选择一种避让策略：

- 同轨道内按日期接近程度自动错层。
- 同一天事件聚合成一个小组卡片，点击展开。
- 限制卡片最小宽度和最大宽度，近距离事件自动压缩。

MVP 建议：

- 同一月份同一轨道最多显示前若干高优先级节点。
- 低优先级节点进入“更多”入口。
- 优先保证赛事和高优先级里程碑可读。

## 5. 赛事事件设计

赛事事件来自后端 Race Resolver 已确认的正式赛事节点。

### 5.1 展示字段

赛事卡片建议包含：

- 赛事名称
- 日期
- 运动类型或赛事类型
- 成绩
- 排名，如有
- 关键距离，如有
- Activity Detail 回跳入口

字段必须来自 Timeline ViewModel 的安全展示字段。

### 5.2 PB 皇冠规则

PB 不再作为独立里程碑节点进入时间轴。若某项赛事本身是 PB，则在赛事卡片上展示皇冠标识。

皇冠规则：

- 人生 PB：右上角显示较大的金色 `👑`
- 当年 PB：右上角显示较大的银色 `👑`
- 非 PB：不显示皇冠

视觉要求：

- 皇冠应压在卡片右上角。
- 尺寸要明显大于普通标签。
- 皇冠允许轻微覆盖卡片边缘，形成高光感。
- 金色和银色不能只靠颜色区分，应在 `title` / `aria-label` 或可访问文本中标注 `人生 PB` / `当年 PB`。

工程边界：

- 前端不得根据成绩、日期、距离自行判断人生 PB 或当年 PB。
- 后端应返回明确字段，例如：

```json
{
  "pb_badge_scope": "career"
}
```

建议枚举：

- `career`：人生 PB
- `season`：当年 PB
- `none` 或空值：非 PB

若同一赛事同时满足人生 PB 和当年 PB，优先展示人生 PB 的金色皇冠。

## 6. 里程碑事件范围

本版里程碑只包含三类：

1. 成就类
2. 累计节点类
3. 首次类

明确不包含：

- PB 里程碑：PB 已经在赛事卡片中表达。
- 记忆类里程碑：暂不做，避免给活动列表增加新的规则判断。
- 普通活动流水：不进入时间轴。

## 7. 成就类里程碑定义

成就类用于表达持续性、挑战性、探索性和年度坚持。它不表达 PB，不复制累计节点，不展示低置信度候选。

成就类节点必须由后端 Achievement Resolver 或 Timeline Milestone Resolver 生成。

### 7.1 稳定训练成就

强调持续训练，不看单次成绩。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `regular_training_4_weeks` | 连续 4 周规律运动 | 连续 4 个自然周，每周至少 3 天有活动 | 达成条件的最后一次活动日期 |
| `regular_training_8_weeks` | 连续 8 周规律运动 | 连续 8 个自然周，每周至少 3 天有活动 | 达成条件的最后一次活动日期 |
| `regular_training_12_weeks` | 连续 12 周规律运动 | 连续 12 个自然周，每周至少 3 天有活动 | 达成条件的最后一次活动日期 |
| `high_frequency_training_month` | 高频训练月 | 单个自然月内运动天数达到 18 天 | 当月第 18 个运动日对应活动日期 |

去重规则：

- 同一连续周期内，如果 4 周、8 周、12 周逐步达成，可以分别生成。
- 同一天如同时达成多个稳定训练成就，Timeline 默认展示最高等级，其他可进入详情或后续“更多”。

### 7.2 爬升挑战成就

强调地形和爬升能力。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `single_elevation_gain_1000m` | 千米爬升挑战 | 单次活动累计爬升 `total_ascent_m >= 1000` | 对应活动日期 |
| `single_elevation_gain_2000m` | 双千米爬升挑战 | 单次活动累计爬升 `total_ascent_m >= 2000` | 对应活动日期 |

去重规则：

- 每个阈值作为生涯成就只生成一次。
- 若同一次活动首次跨过 1000m 和 2000m，Timeline 优先展示 2000m。

### 7.3 高海拔探索成就

强调活动到达的环境高度。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `first_max_altitude_3000m` | 首次踏上海拔 3000m+ | 单次活动最高海拔 `max_alt_m >= 3000`，取首次达成 | 对应活动日期 |
| `first_max_altitude_5000m` | 首次踏上海拔 5000m+ | 单次活动最高海拔 `max_alt_m >= 5000`，取首次达成 | 对应活动日期 |

去重规则：

- 如果首次 5000m+ 活动同时也是首次 3000m+，Timeline 优先展示 `首次踏上海拔 5000m+`。
- 3000m+ 可视为被覆盖，不在同一位置重复展示。

### 7.4 多运动探索成就

强调运动类型丰富度。

有效运动类型应来自后端 canonical sport，不由前端从标题或设备字段推断。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `multi_sport_2_types` | 双栖运动者 | 生涯累计出现 2 种有效运动类型 | 第 2 种运动类型首次出现的活动日期 |
| `multi_sport_3_types` | 多项运动者 | 生涯累计出现 3 种有效运动类型 | 第 3 种运动类型首次出现的活动日期 |
| `multi_sport_5_types` | 全能探索者 | 生涯累计出现 5 种有效运动类型 | 第 5 种运动类型首次出现的活动日期 |

有效运动类型建议：

- 跑步
- 骑行
- 游泳
- 徒步 / 步行
- 力量训练

不建议把设备子类型、室内外细分、通勤等碎片类型纳入 MVP。

### 7.5 年度坚持成就

年度坚持是单年内达成的节点，不等同于生涯累计节点。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `year_activity_100` | 年度 100 次运动 | 某自然年内第 100 次活动 | 第 100 次活动日期 |
| `year_running_distance_1000km` | 年度跑步 1000 km | 某自然年跑步累计首次跨过 1000 km | 跨越阈值的活动日期 |
| `year_cycling_distance_3000km` | 年度骑行 3000 km | 某自然年骑行累计首次跨过 3000 km | 跨越阈值的活动日期 |
| `year_elevation_gain_50000m` | 年度爬升 50000 m | 某自然年累计爬升首次跨过 50000 m | 跨越阈值的活动日期 |

## 8. 累计节点类里程碑定义

累计节点表达生涯累计跨过大阈值的时刻。

核心原则：

- 只显示大节点，不能太密。
- 当某次活动完成后，累计值首次跨过阈值，即在这次活动日期生成节点。
- 同一次活动跨过多个阈值时，Timeline 默认展示最高优先级的一个节点。
- 阈值由后端固定，前端不得判断。

建议阈值：

| 类别 | 事件 key | 阈值 |
| --- | --- | --- |
| 总活动次数 | `total_activity_count_milestone` | 100 / 200 / 300 / 500 / 1000 次 |
| 总里程 | `total_distance_milestone` | 500 / 1000 / 2000 / 5000 / 10000 km |
| 跑步累计里程 | `running_distance_milestone` | 500 / 1000 / 2000 / 5000 km |
| 骑行累计里程 | `cycling_distance_milestone` | 1000 / 3000 / 5000 / 10000 km |
| 累计爬升 | `total_elevation_gain_milestone` | 10000 / 50000 / 100000 m |
| 累计运动时长 | `total_duration_hours_milestone` | 100 / 300 / 500 / 1000 小时 |

卡片建议字段：

- 标题：如 `累计跑步 1000 km`
- 日期：跨越节点的活动日期
- 核心值：如 `1000 km`
- 辅助信息：由哪次活动达成
- 回跳：跨越阈值的活动详情页

## 9. 首次类里程碑定义

首次类表达用户运动生涯中的第一次。

本版取舍：

- 首次类保留普通生涯第一次和标准距离第一次。
- 爬升、高海拔阈值归入成就类，不在首次类重复展示。
- 第一次城市、第一次设备、第一次上传照片、第一次写故事暂不做。

建议事件：

| 事件 key | 标题 | 判定条件 | 触发日期 |
| --- | --- | --- | --- |
| `first_activity` | 第一次运动记录 | 生涯第一条有效 Activity | 对应活动日期 |
| `first_sport_activity` | 第一次跑步 / 第一次骑行等 | 某 canonical sport 第一次出现 | 对应活动日期 |
| `first_race` | 第一场正式赛事 | Race Resolver 首次确认正式赛事 | 对应赛事活动日期 |
| `first_running_5k` | 第一次 5K | 跑步活动首次完成 5K 标准距离 | 对应活动日期 |
| `first_running_10k` | 第一次 10K | 跑步活动首次完成 10K 标准距离 | 对应活动日期 |
| `first_half_marathon` | 第一次半马 | 跑步活动首次完成半马标准距离 | 对应活动日期 |
| `first_marathon` | 第一次全马 | 跑步活动首次完成全马标准距离 | 对应活动日期 |
| `first_cycling_50k` | 第一次 50K 骑行 | 骑行活动首次达到 50 km | 对应活动日期 |
| `first_cycling_100k` | 第一次 100K 骑行 | 骑行活动首次达到 100 km | 对应活动日期 |
| `first_cycling_200k` | 第一次 200K 骑行 | 骑行活动首次达到 200 km | 对应活动日期 |

合并规则：

- 如果同一活动同时满足多个首次，例如 `第一场正式赛事` 和 `第一次半马`，后端可合并成更强标题，例如 `首次半马赛事`。
- Timeline 默认展示合并后的高优先级节点，避免同日重复堆叠。

## 10. Timeline ViewModel 建议

后续任务可以在不暴露原始字段的前提下，扩展 Timeline ViewModel。

建议节点结构：

```json
{
  "id": "timeline-node-xxx",
  "type": "race",
  "subtype": "half_marathon",
  "date": "2026-06-15",
  "year": 2026,
  "month": 6,
  "day": 15,
  "track": "race",
  "title": "青岛半程马拉松",
  "badge": "公路赛",
  "value": "01:22:05",
  "meta": "半程马拉松",
  "pb_badge_scope": "season",
  "priority": 90,
  "detail_link": {
    "activity_id": "123",
    "source": "career_timeline"
  }
}
```

建议字段说明：

| 字段 | 说明 |
| --- | --- |
| `id` | 后端生成的安全节点 ID |
| `type` | `race` 或 `milestone` |
| `subtype` | 具体事件类型，如 `first_half_marathon` |
| `date/year/month/day` | 安全日期展示字段 |
| `track` | 前端轨道建议值：`race` / `milestone` |
| `title` | 用户可见标题 |
| `badge` | 类型标签 |
| `value` | 核心数值 |
| `meta` | 辅助说明 |
| `pb_badge_scope` | 赛事 PB 皇冠：`career` / `season` / `none` |
| `priority` | 同日或拥挤时的展示优先级 |
| `detail_link.activity_id` | 回跳 Activity Detail |

禁止字段：

- `points`
- `points_json`
- `track_json`
- `raw_records`
- `fit_records`
- `file_path`
- `storage_ref`
- 本地绝对路径
- SQLite schema

## 11. 前端实现原则

- 前端只渲染后端返回节点。
- 前端只做布局计算，例如按 `day` 计算横向位置。
- 前端不得读取活动原始字段来判断赛事、PB、成就、首次或累计节点。
- 前端不得从 DOM、标题、距离、日期临时推断事件类型。
- 内容筛选只传参或筛选后端已声明的 `type/track`，不改变事实语义。
- 节点卡片必须保留图标或标签，不能只靠颜色区分。
- 窄窗口下可以退化为纵向月份列表，但不能丢失类型标签和 Activity Detail 回跳。

## 12. 视觉方向

整体参考“深色背景 + 月份纵向分组 + 横向日期定位 + 卡片轨道”。

建议：

- 顶部胶囊：清晰、克制，当前选中状态明显。
- 月份标签：数字月份大，英文月份小，形成时间锚点。
- 赛事卡片：更偏运动档案卡，强调成绩、赛事类型、日期。
- 里程碑卡片：更轻，强调标题和核心值，不抢赛事卡片主视觉。
- PB 皇冠：大而明确，压住赛事卡片右上角。
- 同一月份内的日期轴可以弱化为隐形轴，不一定显示 1-31 每个刻度。

## 13. 建议任务拆解

### ACS-Next-06B-01：Timeline 产品契约与 API ViewModel 扩展

目标：

- 移除 Timeline 对 `years[].season` 展示依赖。
- 定义 `race / milestone` 两类正式节点。
- 为赛事节点补充 `pb_badge_scope`。
- 为节点补充 `day / track / priority` 等布局字段。

目标：

- 时间轴顶部支持 `全部 / 赛事 / 里程碑`。
- 年份胶囊默认 `全部`。
- 月份内赛事按日期横向定位。
- 人生 PB / 当年 PB 使用金色 / 银色皇冠展示。

### ACS-Next-06B-03：里程碑 Resolver MVP

目标：

- 生成成就类、累计节点类、首次类里程碑。
- 排除 PB 里程碑和记忆类里程碑。
- 处理同日、同活动、同类型阈值去重。

### ACS-Next-06B-04：里程碑轨道与卡片避让

目标：

- 月份内第二行展示里程碑。
- 处理同日/近日期卡片拥挤。
- 补齐空态、更多入口和窄屏退化。

### ACS-Next-06B-05：Timeline 视觉验收与性能收口

目标：

- 验证 500+ 节点下渲染不卡顿。
- 验证桌面宽屏布局、窄屏布局、年份切换、类型切换。
- 补充前端静态契约测试和必要后端 Resolver 测试。

## 14. 验收标准

- 时间轴默认显示 `全部年份 + 全部内容`。
- 用户可以切换具体年份。
- 用户可以切换 `全部 / 赛事 / 里程碑`。
- 同一个月份内，赛事和里程碑分轨道展示。
- 卡片按日期横向落位。
