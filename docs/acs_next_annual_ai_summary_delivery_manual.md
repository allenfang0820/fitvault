# ACS 年度 AI 总结开发交付手册

## 1. 文档目的

本文冻结 ACS 年度 AI 总结的产品目标、交互规则、数据契约、AI 边界、持久化策略、更新条件和验收标准，作为后续任务拆解、编码、测试与 review 的最终功能指导文件。

若本文与项目总契约冲突，优先级如下：

1. `Activity` 单一事实源与 Resolver 语义边界。
2. `docs/脉图运动生涯系统（ACS）开发团队交付手册.md`。
3. 本交付手册。
4. `docs/js_api_contract.json` 中已登记的现行 API 契约；开发时必须同步更新为本手册冻结的年度能力。
5. 历史 Phase7 完成报告和旧测试描述。

历史 Phase7 文档仍可能出现 `representative_memories`、`memory_count` 或旧 MemoryItem 能力。这些内容已经被 2026-07-13 的“记忆能力退役”决策覆盖，不得进入年度 AI Snapshot 或年度报告。

## 2. 当前基线

当前仓库已经具备：

- Overview 年度卡片和年度统计 ViewModel。
- 独立的“年度 AI 总结”二级页面。
- 全生涯 `Career Snapshot` 白名单骨架。
- `career_snapshots` 本地持久化表。
- 年度总结页加载、空态和错误态渲染。

2026-07-15 产品收敛后，全生涯“生涯总结”页面和模式切换已经退役。`generate_career_insight` 与 `get_latest_career_snapshot` 仅作为无前端入口的兼容接口保留，全生涯 Snapshot 不再构成用户功能。

当前尚未具备：

- 按年份隔离的 `Year Snapshot`。
- 年度 Snapshot 数据指纹。
- 真实 LLM 年度总结。
- 年度 AI 输出持久化和缓存命中策略。
- “未生成 / 已是最新 / 数据已更新”状态机。
- 年度卡片到指定年份 AI 总结页的导航。
- 仅在年度输入事实发生变化时允许更新报告的规则。

因此，本功能不是给现有 fallback 文案增加一个年份参数，而是建立一条独立、可追溯、可缓存、可更新的年度报告链路。

## 3. 产品定位

年度 AI 总结用于回答：

```text
这一年，我的运动生涯发生了什么？
```

它是一份有后端事实证据的年度回顾，不是：

- 活动流水列表。
- 单次运动复盘集合。
- 自动训练计划。
- 医疗、伤病或健康诊断。
- AI 自行判断赛事、PB、成就或里程碑的入口。
- 对年份进行“高光年 / 赛事年 / 记录年”等等级评价。

年度报告应把可信年度事实组织为一篇从头到尾可以连续阅读的年度故事，帮助用户回答：这一年付出了什么、取得了什么成绩、参加了什么比赛、发生了什么进步、哪些时刻值得记住，以及想对下一年的自己说什么。

用户可见的最终产物不是若干互不相连的分析字段或卡片，而是一篇完整报告。结构化 JSON 只是后端校验、事实回填、页面编排和未来生成分享图片的内部母稿。

### 3.1 v3 叙事节奏升级

v3 的核心目标是让报告更像用户愿意读完、愿意分享的年度运动故事，而不是统计摘要。

必须遵守：

- 开篇先给年度感受、主线或阅读入口，不得在第一句话一次性公布活动次数、总里程、总时长、赛事、PB 和成就等全部年度数据。
- 年度事实必须逐层揭示：先说明这一年如何被记录下来，再逐步展开活动次数、里程、时长、赛事、PB、成就、城市和节奏。
- 每组数字都必须服务一个叙事问题：它意味着什么，为什么值得被这一年记住。
- 报告必须从后端可信线索中选择少量高光时刻，包括赛事、PB、成就、最长距离、最高海拔、累计爬升或城市足迹等；没有对应事实时自然省略。
- AI 不得自行判定高光、PB、赛事、城市或里程碑；只能解释后端提供的线索。
- 城市内容允许进入年度故事，但必须来自活动定位事实，并使用受控城市文化词典。AI 不得自由使用常识扩写，更不得暗示用户实际吃过、旅行过或发生过 Snapshot 外的生活事件。

### 3.2 v4 分享欲与成就感语气

v4 在 v3 事实边界不变的基础上，调整报告语气目标：

```text
用户读完后应产生：这一年我真的做成了一些事，值得发出来，也值得被自己看见。
```

因此，年度报告不得写成过于平实的审计报告。允许：

- 有成就感：完成、跨过、点亮、抵达、留下坐标、把运动带去更多地方。
- 有分享欲：适合截图分享的标题、副标题、收束句和年度主线。
- 有郑重感：赛事、PB、首次完成、最长距离、最高海拔、新城市等高光应被认真庆祝。
- 有画面感：围绕城市、赛事和高光组织叙事，但只写后端事实允许的画面。

仍然禁止：

- “封神”“王者”“炸裂”“逆天”等营销鸡血词。
- “含金量不言而喻”“硬核”“彻底爆发”等短视频式夸张表达。
- 未经事实支持的“春节假期”“旅行”“身体感受”“看世界的方式”“吃了火锅/和牛”等生活和心理推断。
- 用年度等级评价用户，例如“高光年”“失败年”“最强一年”。

理想语气不是把情绪压平，而是：

```text
有光、有分量、愿意庆祝，但每一句都站在后端事实上。
```

推荐阅读节奏：

1. 开篇：先写这一年的感觉、持续或回望入口，不报完整成绩单。
2. 积累：逐步展开活动次数、里程和时长，每次只出现一组核心事实。
3. 高光：进入赛事、PB、里程碑、最长距离、最高海拔等可信节点。
4. 足迹：写城市覆盖和有代表性的城市记忆。
5. 节奏：回看月份与阶段变化。
6. 比较：仅在后端同期比较可用时出现。
7. 收束：写给下一年的自己。

## 4. 核心产品决策

### 4.1 年度卡片是入口，不是生成按钮

Overview 年度卡片始终可点击。

点击年度卡片只执行：

```text
进入 AI 总结页 -> 切换到年度总结模式 -> 选中对应年份
```

点击年度卡片不得直接调用 LLM，不得直接刷新或覆盖既有报告。

年度卡片悬停提示统一为：

```text
查看年度总结
```

不得继续使用：

```text
点击我试试
```

原因：年度报告可能处于未生成、已是最新、可更新或不可用等不同状态，年度卡片不承担生成状态判断。

### 4.2 报告按年份沉淀

每个有运动数据的自然年拥有独立年度报告。

- 首次生成后持久化到本地 AI 输出缓存。
- 再次进入对应年份时优先读取缓存，不重复调用 AI。
- 年份之间互不覆盖。
- 年度报告只在年度之间独立沉淀，不再提供全生涯总结模式。
- 用户可在 AI 总结页切换年份查看已经沉淀的报告。

### 4.3 是否允许更新只由年度事实变化决定

不得根据“当年 / 过往年度”决定能否刷新。

任意年份只要进入年度 Snapshot 的事实发生变化，对应报告就进入“可更新”状态。

例如：

- 2026 年新增一条活动，2026 报告可更新。
- 2026 年编辑活动标题或删除活动，2026 报告可更新。
- 后来补导入 2022 年活动，2022 报告可更新。
- 赛事、PB、成就 Resolver 结果变化，只影响事件所属年份。

未影响年度 Snapshot 的变化不得触发更新，例如：

- 赛事照片增加、删除或排序。
- UI 设置变化。
- 本地媒体预览变化。
- 不进入年度报告的 debug 字段变化。

### 4.4 无数据变化时不提供重复生成

报告对应的 Snapshot 指纹与当前年度 Snapshot 指纹一致时：

- 状态显示“已是最新”。
- 不显示“重新生成”或“换一种说法”按钮。
- 不允许通过前端参数绕过该限制重复消耗 AI。

## 5. 页面信息架构

### 5.1 顶部年份

年度 AI 总结页直接展示年份胶囊，不再提供模式切换：

```text
2026 | 2025 | 2024 | ...
```

规则：

- 从年度卡片进入时选中卡片年份。
- 从顶部“AI 总结”导航进入时，默认选中最近一个有活动数据的年份。
- 年份列表来自后端，不由前端扫描活动或 DOM 推断。
- 没有任何年度数据时，不展示伪造年份。

### 5.2 年度报告页面结构

年度报告必须按一篇完整长文编排，而不是把 `主线 / 节奏 / 比较 / 方向` 渲染成互相割裂的工程字段卡片。推荐阅读顺序：

1. 年度标题与一句副标题。
2. AI 开篇：先建立年度主线，不一次性报完全部数据。
3. 分层事实引导：活动、里程、时长、赛事/PB/成就、城市等事实按阅读节奏逐步展开。
4. 年度主线：串联坚持、主要运动和阶段变化。
5. 高光时刻：赛事、PB、成就、海拔/距离/爬升等后端可信节点；没有对应事实时自然省略。
6. 运动足迹：覆盖城市和有代表性的城市记忆；只使用受控城市文化词典。
7. 年度节奏：按月度事实描述积累、活跃变化和阶段，不推断生活原因。
8. 与上一年相比：仅在后端比较可用时出现。
9. 收束段与“写给下一年的你”。
10. 生成时间、数据截止信息和 AI 免责声明。

报告章节可通过留白、细分隔线、引文和证据节点形成阅读节奏，但正文必须在视觉上保持连续；除可回跳的赛事/PB/活动证据外，不把每个段落包成独立卡片。

### 年份标题

- 年份数字使用与年度卡片一致的 DIN Next 字体。
- 当前年份必须明确为阶段性总结，不得伪装成完整自然年。
- 显示 `generated_at` 和 `data_through`。`generated_at` 的数据库审计值统一保存 UTC，API/ViewModel 和页面必须转换为运行设备当地时间后展示，并保留明确时区偏移。

### 年度事实概览

事实概览由后端直接渲染，不来自 AI 文本：

- 活动次数。
- 总里程。
- 总时长。
- 赛事数量。
- PB 数量。
- 成就 / 里程碑数量。
- 可靠时可展示覆盖城市数量。

### 分层事实引导

分层事实引导由后端根据 Year Snapshot 生成，不由 AI 复述或计算。它取代 v2 的单句 `fact_lead`，避免开篇第一句话一次性公布全部年度数字。

允许：

```text
直到把这一年的记录放在一起，才会发现：这样的普通日子，已经积累成了 128 次运动。
这些活动带你走过了 1246 公里。它不只是一个总数，也记录了你如何把运动慢慢留在生活里。
```

仅输出 Snapshot 中真实存在且适合用户阅读的事实。数字格式、单位和当前部分年度表述由后端负责。

禁止：

```text
这一年，你完成了 128 次运动，累计 1246 公里，运动了 86 小时，参加了 3 场比赛，获得 2 项 PB。
```

### 这一年的主线

AI 用一段短文概括年度运动轨迹，重点描述：

- 主要运动类型。
- 活跃节奏。
- 赛事、PB 或重要里程碑构成的年度主线。
- 当前数据不足时的克制表达。

不得输出年度等级、虚构身份或未经后端确认的因果判断。

### 关键时刻

展示 3 至 5 个后端证据节点：

- 赛事。
- PB。
- 成就或里程碑。
- 后端确定的代表活动。
- 单次最长距离。
- 单次最长时长。
- 单次最高海拔或最大爬升。
- 年度累计爬升突破受控阈值。
- 首次点亮城市或代表性城市足迹。

每个关键时刻必须包含 `activity_id` 并可回跳 Activity Detail。AI 只能选择证据和补充简短解读，不能生成事件标题、日期、成绩或赛事身份。

高光排序原则：

1. 赛事、PB 和高分成就优先。
2. 年度内首次或新纪录优先。
3. 突破受控阈值优先，例如单次最高海拔超过 5000 m、年度累计爬升超过 5000/10000/20000/50000 m。
4. 城市足迹最多 2 条，避免把报告写成旅行清单。
5. 总数最多 5 条；证据不足时允许 0 至 2 条。

### 城市足迹与受控文化词典

城市事实必须来自 Activity 安全字段或 Resolver evidence，允许字段包括城市名、活动数、首次日期、代表活动和受控文化提示。

受控文化提示示例：

- 成都：火锅。
- 神户：和牛。
- 北京：城墙、胡同或中轴线。
- 上海：江边、城市夜色。
- 杭州：西湖。

AI 可以写：

```text
成都因火锅闻名，而你也在这座城市留下了自己的运动坐标。
```

AI 不可以写：

```text
你跑完以后吃了一顿火锅。
```

除非后端事实明确提供，否则不得写用户吃了什么、去了哪里游玩、和谁同行、为什么来到这座城市。

### 运动节奏

以年度月度摘要为基础，描述：

- 活跃月份分布。
- 主要运动类型变化。
- 后端可确认的阶段性增长、中断或恢复。

不得从缺少活动直接推断受伤、懈怠、情绪或生活事件。

### 与上一年相比

比较结果必须由后端计算。

- 已结束年份：完整自然年对完整自然年。
- 当前年份：当年截至最新进入 Year Snapshot 的事实日期，对比上一年相同日期范围。
- 上一年数据不足时，返回 `unavailable`，不强行生成增减结论。
- AI 只解释后端提供的差值，不自行计算百分比。

### 收束与写给下一年的你

最多输出 2 至 3 条方向性建议。

允许：

- 继续保持某类稳定习惯。
- 根据已确认的年度节奏提出温和方向。
- 建议关注数据连续性或目标选择。

禁止：

- 生成详细训练计划。
- 医疗、伤病诊断。
- 强制性运动处方。
- 超出 Snapshot 证据的能力评价。

收束段应先回答“这一年真正值得记住的是什么”，再给出一段写给下一年的话。语气温暖、克制、真诚，不制造焦虑，也不要求下一年必须超过这一年。

## 6. 页面状态机

每个年份必须明确处于以下状态之一：

| 状态 | 含义 | 页面行为 | 主操作 |
| --- | --- | --- | --- |
| `no_data` | 该年份无可总结活动 | 展示稳定空态 | 无 |
| `not_generated` | 有年度数据但从未生成报告 | 展示事实概览与待生成态 | 生成年度总结 |
| `ready` | 报告指纹与当前 Snapshot 一致 | 展示沉淀报告 | 无 |
| `stale` | 已有报告，但年度 Snapshot 已变化 | 保留旧报告并提示有新数据 | 更新年度总结 |
| `generating` | 正在生成或更新 | 保留事实概览；更新时保留旧报告 | 禁用重复操作 |
| `failed` | 本次生成失败 | 保留旧报告或事实概览 | 重试本次生成 |
| `ai_unavailable` | AI 配置缺失或服务不可用 | 展示事实概览和已有报告 | 进入配置或稍后重试 |

### 6.1 未生成态

在任何 AI 报告生成前，AI 总结页不得空白。

必须展示：

- 年度模式和年份选择。
- 后端年度事实概览。
- `年度总结尚未生成` 状态。
- `生成 {year} 年度总结` 按钮。
- 当前年份的数据周期信息。
- 可选的后端关键事件预览。

不得展示伪装成 AI 结果的固定叙事。

### 6.2 可更新态

当年度事实变化时：

- 旧报告继续可读。
- 状态显示“有新的运动数据”。
- 显示“更新年度总结”。
- 更新过程中不得清空旧报告。
- 更新成功后原子切换为新报告。
- 更新失败后继续保留旧报告。

## 7. Year Snapshot 契约

### 7.1 Snapshot 定位

`Year Snapshot` 是 AI 唯一允许消费的年度上下文。

数据流：

```text
Activity / Resolver canonical facts
  -> Year Snapshot Builder
  -> Snapshot 白名单与稳定排序
  -> source_fingerprint
  -> AI Prompt Assembler
  -> LLM
  -> 输出校验与证据回填
  -> 年度报告缓存
  -> AI 总结页 ViewModel
```

禁止数据流：

```text
LLM -> SQLite 查询
LLM -> raw FIT
LLM -> points / track_json
LLM -> 本地文件或照片路径
LLM -> canonical 事实表写入
前端 -> 自行拼装年度 Snapshot
```

### 7.2 建议 Snapshot 结构


### 7.3 Snapshot 允许字段

允许：

- 年度安全聚合统计。
- 按运动类型的安全聚合。
- 月度聚合摘要。
- Resolver 已确认的赛事、PB、成就和里程碑摘要。
- 用于 Activity Detail 回跳的 `activity_id`。
- 后端确定的比较差值。
- 数据质量和缺失原因。

禁止：

- raw FIT。
- `points`、`points_json`、`track_json`。
- `file_path`、`storage_ref`、本地绝对路径、`file://`。
- SQLite schema、表名或 SQL。
- 照片、缩略图、媒体引用。
- 已退役的通用记忆、故事或 `representative_memories`。
- 用户 token、LLM 配置、Provider 授权信息。

## 8. 数据指纹与更新判断

### 8.1 指纹生成规则

`source_fingerprint` 必须基于经过白名单裁剪、稳定排序后的年度 Snapshot 事实内容计算。

建议：

```text
source_fingerprint = sha256(canonical_json(report_source_fields))
```

计算前必须：

- 字典键稳定排序。
- 事件列表按 `date + type + evidence_id` 稳定排序。
- 排除 `generated_at`、`as_of_date`、状态文案、traceId 等非事实字段。
- 排除 UI 展示顺序之外的随机字段。
- 数字统一精度，避免浮点格式变化造成无意义刷新。

当前日期自然推进但没有新增、编辑、删除活动或 Resolver 事实变化时，fingerprint 必须保持不变。当前年份的同期比较截止点使用 `data_through`，不得使用每天变化的系统日期。

不得仅使用：

- 数据库 `MAX(updated_at)`。
- 活动总数。
- 当前时间。
- 文件修改时间。

这些值不能准确表达“进入报告的事实是否变化”。

### 8.2 报告状态判断

```text
没有年度数据
  -> no_data

有年度数据，没有任何成功报告
  -> not_generated

成功报告 fingerprint == 当前 fingerprint
  -> ready

成功报告 fingerprint != 当前 fingerprint
  -> stale
```

## 9. AI 输出契约

### 9.1 AI 只输出结构化长文母稿

AI 返回结构化母稿，后端将其与事实导语、证据事实和元数据拼接为一篇完整报告：

```json
{
  "schema_version": "acs.year.report.v3",
  "year": 2026,
  "title": "2026，我把运动稳稳地留在了生活里",
  "subtitle": "稳定积累中，也完成了几次清楚的突破",
  "opening": "这些数字不是突然出现的，它们来自一个个普通日子里的出门、坚持和回来。",
  "body_sections": [
    {
      "type": "annual_story",
      "heading": "这一年的主线",
      "paragraphs": ["..."],
      "evidence_ids": []
    },
    {
      "type": "races",
      "heading": "你完成的比赛",
      "paragraphs": ["..."],
      "evidence_ids": ["race:123"]
    },
    {
      "type": "progress",
      "heading": "看得见的进步",
      "paragraphs": ["..."],
      "evidence_ids": ["pb:456"]
    },
    {
      "type": "rhythm",
      "heading": "这一年的节奏",
      "paragraphs": ["..."],
      "evidence_ids": []
    }
  ],
  "closing": "所以，这一年真正值得记住的，不只是某一个成绩……",
  "letter_to_next_year": "写给下一年的你：继续把运动留在生活里。",
  "share_caption": "我没有突然变强，但我一直在回来。",
  "caveats": []
}
```

`body_sections.type` 仅允许：`annual_story`、`races`、`progress`、`rhythm`、`comparison`。章节按上述顺序输出；没有赛事、进步证据或可靠同比时，对应章节必须省略，不得用空洞文案补位。`annual_story` 和 `rhythm` 为基础章节。

`share_caption` 是未来社交媒体分享图片的短文案预留，不得包含 AI 自行复述的数字、成绩或日期；分享图片不在本期实现范围。

AI 不返回：

- 自己生成的活动标题、日期、成绩、赛事类型。
- 自己计算的统计数字或百分比。
- 在叙事正文中自行复述可能被误认为 canonical 的精确数字、日期或成绩；这些由后端事实导语和证据节点呈现。
- `detail_link`。
- HTML、Markdown 代码块或脚本。
- 数据库字段、路径或内部实现说明。

### 9.2 证据回填

后端收到 AI 输出后必须：

1. 校验 JSON schema。
2. 校验 `year` 与请求年份一致。
3. 校验所有 `evidence_id` 存在于 Snapshot `evidence_catalog`。
4. 去除重复和未知证据。
5. 限制关键时刻数量。
6. 按章节类型校验其证据类型和是否应当存在；无赛事、PB/成就或同比事实时删除对应章节。
7. 使用后端证据重新填充标题、日期、成绩和 `detail_link`。
8. 由后端生成事实导语，并与 AI 母稿拼成统一报告 ViewModel。
9. 对输出文本做长度限制和安全清洗；AI 自行复述精确数字、日期或成绩的句子必须删除，不能进入用户可见正文。

前端不得直接信任 LLM 返回的事件事实。

### 9.3 Prompt 约束

Prompt 必须明确：

- 只能使用给定 Year Snapshot。
- 不推断伤病、心理、生活事件或训练动机。
- 不生成年度等级标签。
- 不生成训练计划或医疗建议。
- 当前年份必须使用“截至当前数据周期”的表达。
- 数据不足时主动降级，不补全不存在的故事。
- 输出严格 JSON，不附加解释文本。
- 最终读感是一篇连续的年度故事，不是数据分析表或字段解释。
- 语气温暖、克制、真诚、有鼓励感，像懂运动数据的朋友陪用户回望一年。
- 承认努力，也尊重普通年份；鼓励用户看见持续，不只崇拜 PB。
- 禁止“封神”“王者”“炸裂”“逆天”等营销式、鸡血式或制造焦虑的表达。
- 禁止把没有比赛、没有 PB 或活动较少的年份写成失败。

Prompt 需要独立版本号，例如：

```text
acs.year.summary.zh-CN.v2
```

Prompt 版本变化不得自动让所有报告进入“年度数据已更新”状态。是否允许重新生成新 Prompt 版本需要独立产品决策；本期只由 Snapshot 事实变化触发用户可见更新。

## 10. 持久化设计

### 10.1 年度 Snapshot

可复用现有 `career_snapshots` 表：

```text
id = career_snapshot:year:{year}
snapshot_type = career_year
source_version = acs.year.v1
content_json = 白名单 Year Snapshot
```

年度 Snapshot 是可重新构建的派生数据，不是 canonical 事实源。

### 10.2 AI 输出缓存

新增独立表，建议命名：

```text
career_ai_insights
```

建议字段：

```text
id
scope                 # year / career
scope_key             # 年度为 2026
snapshot_fingerprint
snapshot_version
prompt_version
model_id
content_json
generated_at
created_at
status                # ready / superseded
```

建议唯一约束：

```text
scope + scope_key + snapshot_fingerprint + prompt_version + model_id
```

策略：

- 成功输出才进入缓存。
- 同一指纹重复请求命中缓存，不重复调用 AI。
- 新指纹生成成功后，新报告成为当前展示版本。
- 历史记录可保留用于审计，但第一版不提供用户可见版本管理。
- AI 输出不得写入 Activity、Race、PB、Achievement 等 canonical 表。
- `generated_at`、`created_at`、`updated_at` 持久化为 UTC；缓存排序和审计使用 UTC 原值，用户界面不直接展示 UTC 原字符串。

## 11. API 契约建议

年度报告使用独立年度作用域 API。已退役的全生涯 `generate_career_insight` 不得恢复或复用。

### 11.1 获取年度报告页

```text
get_career_year_insight({ year })
```

只读，不调用 LLM。

建议返回：

```json
{
  "year": 2026,
  "available_years": [2026, 2025, 2024],
  "report_state": "stale",
  "can_generate": false,
  "can_refresh": true,
  "has_source_changes": true,
  "facts": {},
  "report": {},
  "generated_at": "2026-07-10T12:00:00+08:00",
  "data_through": "2026-07-11",
  "status": {
    "schema_ready": true,
    "data_ready": true,
    "message": "年度总结有新的运动数据"
  }
}
```

`data_through` 表示最新进入 Year Snapshot 的事实日期，通常等于该年度最新有效 Activity 或派生事件日期。`as_of_date` 只表示页面查看或 Snapshot 构建日期，不参与 fingerprint，也不能单独触发报告更新。已结束年份可另用 `is_partial_year=false` 表示完整自然年范围。

### 11.2 生成或更新年度报告

```text
generate_career_year_insight({ year })
```

规则：

- `not_generated` 时允许生成。
- `stale` 时允许更新。
- `ready` 时直接返回缓存或明确的“已是最新”，不得再次调用 AI。
- `no_data` 时返回稳定验证错误或不可生成状态。
- 前端不得提交 prompt、Snapshot、模型名或任意事实字段。
- 模型配置由现有受控 LLM 配置链路读取。

所有 pywebview API 继续使用统一 envelope：

```text
{ ok, code, msg, data, traceId }
```

## 12. 前端交互契约

### 12.1 年度卡片

年度卡片应改为可访问的原生按钮或等价可键盘操作元素。

必须：

- `aria-label="查看 2026 年度总结"`。
- Enter / Space 可触发。
- 点击只切换页面和年份。
- 悬停提示为“查看年度总结”。
- 保留现有轻微上浮效果。

禁止：

- 在年度卡片上直接调用 LLM。
- 根据前端缓存猜测 `can_refresh`。
- 在卡片上堆叠生成状态胶囊，破坏年度卡片当前克制布局。

### 12.2 AI 总结页操作

主操作严格由后端状态驱动：

| report_state | 按钮 |
| --- | --- |
| `not_generated` | 生成年度总结 |
| `stale` | 更新年度总结 |
| `ready` | 不显示生成按钮 |
| `no_data` | 不显示生成按钮 |
| `generating` | 禁用并显示生成中 |
| `failed` | 重试 |
| `ai_unavailable` | 不显示虚假生成按钮 |

前端不得用“当前年份”判断是否显示刷新按钮。

### 12.3 生成过程

- 首次生成时保留年度事实概览。
- 更新时保留旧报告。
- 首次生成时展示明确的过渡区，主提示为“正在整理你的 {year} 年运动故事”。
- 过渡区按时间轮换展示“正在读取年度活动数据 / 正在整理比赛和 PB / 正在梳理这一年的运动节奏 / 正在生成年度报告 / 正在校验证据来源”；这些是 UI 进度提示，不伪装成后端真实百分比。
- 标题、正文段落和证据节点使用轻量 skeleton 或 shimmer，且必须尊重 `prefers-reduced-motion`。
- 生成按钮进入禁用态并保持可见，避免用户无法判断点击是否生效。
- 同一年份只允许一个生成请求进行中。
- 切换年份不得把 A 年的结果写入 B 年页面。
- 页面销毁或切换后，晚到响应必须通过请求年份和 request token 校验。

## 13. 视觉设计原则

- 延续 ACS 安静、克制、工作型界面。
- 不制作营销式 Hero。
- 年份是第一视觉信号，但不得占满首屏。
- 年度事实使用紧凑、可扫描的无嵌套布局。
- 不把所有段落做成卡片套卡片。
- 页面主体使用文章式单列阅读宽度；事实概览是文章导语的一部分，章节之间依靠排版层级与留白连接。
- 关键时刻可使用重复小卡片，并保持 Activity Detail 回跳。
- AI 文本与事实数据在视觉上分层，避免用户把 AI 叙事误认为 canonical 指标。
- 不使用大面积紫色渐变、装饰光球或纯氛围背景。
- 移动端单列，按钮、年份胶囊和长中文文本不得溢出。
- Loading、旧报告和错误状态不得互相覆盖。

## 14. 安全与隐私边界

年度 AI 功能必须继续遵守：

- AI 只消费 Year Snapshot。
- 不把 Snapshot 原文或 debug JSON 展示给前端用户。
- 不把本地路径、媒体引用、FIT 原文、轨迹点或数据库结构发送给 AI。
- 不在日志中输出完整 Prompt、完整 Snapshot 或完整 AI 响应。
- 日志只保留 request id、year、fingerprint 前缀、prompt version、model id、耗时和结果状态。
- API 错误不得回显 LLM token、URL 密钥或底层请求体。
- AI 输出只能写入独立派生缓存。

## 15. 并发、幂等与失败策略

### 15.1 单飞控制

同一 `year + fingerprint` 同时只能存在一个生成任务。

重复点击：

- 复用进行中的任务状态，或返回“正在生成”。
- 不发起第二次 LLM 请求。

### 15.2 缓存优先

生成前再次检查缓存，避免两个窗口或重复请求产生重复调用。

### 15.3 输出失败

以下情况不得覆盖旧报告：

- 网络失败。
- 超时。
- LLM 返回非 JSON。
- schema 校验失败。
- evidence_id 不合法。
- 输出为空或超过长度限制。

允许对纯格式错误做一次受控修复或重试；不得无限重试。

## 16. 本地 fallback

真实 AI 不可用时，年度页面仍应可使用。

本地 fallback 可以展示：

- 年度事实概览。
- 后端生成的关键事件列表。
- 后端确定的月度节奏摘要。
- 后端计算的同比差值。

本地 fallback 不得伪装成 AI 报告，不得写入 `career_ai_insights` 的成功 AI 缓存。

已有成功报告时，即使 AI 当前不可用，也应继续展示已沉淀报告。

## 17. 数据更新触发矩阵

| 数据变化 | 对应年度 fingerprint | 报告状态 |
| --- | --- | --- |
| 新增 Activity | 变化 | `stale` |
| 删除 Activity | 变化 | `stale` |
| 修改进入 Snapshot 的 Activity 字段 | 变化 | `stale` |
| Race Resolver 结果变化 | 变化 | `stale` |
| PB Resolver 结果变化 | 变化 | `stale` |
| Achievement / Milestone 变化 | 变化 | `stale` |
| 赛事照片增删或排序 | 不变化 | 保持 `ready` |
| UI 样式或设置变化 | 不变化 | 保持 `ready` |
| Snapshot 生成时间变化但事实不变 | 不变化 | 保持 `ready` |
| Prompt 版本变化 | 本期不作为用户可见数据更新 | 保持原状态 |

## 18. 兼容与迁移

### 18.0 v1 / v2 年度报告迁移

- 已缓存的 `acs.year.report.v1` 必须继续可读，不得删除或覆盖。
- 前端通过兼容适配器把 v1 报告映射为旧内容兼容视图，避免数据库迁移失败导致空白。
- `acs.year.report.v2` 是内容形态重大升级，不等同于日常 Prompt 微调。若当前指纹只有 v1 报告，后端可以返回 `format_upgrade_available=true`，页面显示“升级年度故事”。
- `acs.year.report.v3` 是 v4 语气升级后的当前结构化文章 schema，用于把“分享欲与成就感”作为受控格式升级沉淀。若当前指纹只有 v1/v2 报告且 AI 可用，后端可以返回 `format_upgrade_available=true`，页面复用“升级年度故事”动作。
- 格式升级不伪装成“年度数据已更新”，不改变 `source_fingerprint`，也不允许前端传入模型、Prompt、Snapshot 或强制参数。
- v3 生成成功后原子切换当前展示版本；失败继续展示旧报告。
- v3 已存在且指纹未变化时仍禁止重复生成。

### 18.1 全生涯总结退役

- 总览页承担全生涯概览，不再提供独立“生涯总结”模式。
- 前端不得调用 `generate_career_insight` 或 `get_latest_career_snapshot`；接口只为兼容和诊断保留。
- 前端不得保留生涯总结模式切换、刷新按钮、ViewModel 或加载函数。
- `career_snapshots` 与内部 Career Snapshot 工具可为兼容和年度持久化保留，但不得重新形成用户可见的全生涯总结入口。

### 18.2 现有年度卡片

- 保留年度卡片统计胶囊、DIN 年份和悬停上浮。
- 替换悬停文案。
- 增加导航行为和键盘可访问性。
- 不在卡片上增加“未生成 / 可更新”状态胶囊。

### 18.3 旧 Phase7 文档

实施年度 AI 功能时必须同步修正现行任务清单中的过期描述：

- 移除 `representative_memories` 作为 AI Snapshot 白名单。
- 移除 `memory_count` 和旧通用 MemoryItem 语义。
- 明确年度 Snapshot 与内部全生涯 Snapshot 分离，且后者不再对应用户页面。

历史完成报告可保留，但需要标注历史状态，不得作为现行 API 契约。

## 19. 测试要求

### 19.1 Snapshot 测试

- 只包含目标年份数据。
- 删除活动不进入 Snapshot。
- 赛事、PB、成就只包含所属年份的 active 结果。
- 当前年份 period 为部分年度。
- 同期比较使用上一年相同日期范围。
- 禁止字段递归检查。
- 稳定排序和浮点精度一致。
- 相同事实生成相同 fingerprint。
- 任一允许事实变化后 fingerprint 变化。
- 照片变化不改变 fingerprint。

### 19.2 持久化与缓存测试

- 每年 Snapshot 独立保存。
- 每年报告独立缓存。
- 相同 fingerprint 命中缓存，不调用 LLM。
- 新 fingerprint 进入 `stale`。
- 更新成功后切换到新报告。
- 更新失败保留旧报告。
- AI 输出不写 canonical 表。

### 19.3 API 测试

- year 参数白名单和范围校验。
- 统一 pywebview envelope。
- `ready` 状态禁止重复调用 AI。
- `no_data` 不允许生成。
- 未知 payload 字段拒绝。
- AI 不可用时返回稳定状态。
- 不泄露 Snapshot、Prompt、路径或配置。

### 19.4 前端测试

- 年度卡片悬停提示为“查看年度总结”。
- 年度卡片点击只切换 AI 总结页和年份。
- 年度卡片支持键盘操作。
- 未生成态展示事实和生成按钮。
- ready 状态不展示刷新按钮。
- stale 状态展示旧报告和更新按钮。
- 更新过程保留旧报告。
- 切换年份不会串写响应。
- AI 不可用时局部降级。
- 移动端不溢出、不重叠。

### 19.5 真实数据验收

至少覆盖：

- 当前年份有持续新增活动。
- 过往年份补导入活动后可更新。
- 过往年份无数据变化时保持已是最新。
- 有赛事、PB、成就的丰富年度。
- 只有普通活动的轻量年度。
- 只有一种运动类型的年度。
- 多运动类型年度。
- 上一年无可比较数据。
- AI 配置缺失、超时、返回格式错误。

## 20. 验收标准

满足以下条件后，可判定“年度 AI 总结代码闭环完成”：

- 年度卡片可进入指定年份 AI 总结页。
- 年度卡片不直接调用 AI。
- Year Snapshot 白名单与禁止字段测试通过。
- 每个年份拥有稳定 source fingerprint。
- 报告按年份持久化并可再次读取。
- `not_generated / ready / stale` 状态判断正确。
- 只有 `not_generated` 或 `stale` 允许调用 AI。
- 关键事件全部可追溯到后端 evidence 和 Activity Detail。
- 旧报告在更新失败时不丢失。
- 全生涯总结入口和公开 API 已移除，年度链路不依赖其 fallback。
- API、前端和文档契约一致。
- 定向测试全绿。

以下条件未完成前，不得标记更高等级完成：

- 未完成真实 LLM 调用验证，不得标记真实 AI 完成。
- 未完成真实用户数据人工验收，不得标记产品验收完成。
- 未完成 macOS / Windows 打包产物验证，不得标记打包级完成。
- 未完成 Windows 真机验证，不得标记跨平台完成。

## 21. 明确不做

本轮不做：

- 用户编辑 AI 报告正文。
- 用户可见的报告历史版本管理。
- 社交分享海报。
- PDF 导出。
- 自动生成详细训练计划。
- AI 对话追问。
- AI 记忆系统。
- 照片理解或多模态年度总结。
- 在年度卡片中直接展示 AI 结论。
- 因模型或 Prompt 升级批量重写历史报告。

## 22. 推荐实施顺序

1. `ACS-Year-AI-00`：冻结本手册、修正 Phase7 过期 Snapshot 契约。
2. `ACS-Year-AI-01`：实现 Year Snapshot 白名单与同期比较。
3. `ACS-Year-AI-02`：实现稳定 fingerprint 和报告状态机。
4. `ACS-Year-AI-03`：实现年度 Snapshot 持久化与 AI 输出缓存表。
5. `ACS-Year-AI-04`：实现年度报告只读 API 和本地 fallback。
6. `ACS-Year-AI-05`：实现年度卡片导航、年度模式和页面状态。
7. `ACS-Year-AI-06`：接入真实 LLM Prompt、JSON schema 和 evidence 回填。
8. `ACS-Year-AI-07`：实现更新、单飞、缓存命中和失败保留。
9. `ACS-Year-AI-08`：完成真实数据验收、视觉 review 和跨平台代码级检查。

每个任务开始前必须刷新项目契约意识：默认使用首次阅读形成的项目契约摘要；只有当前任务、改动文件、测试失败或 review 风险需要时，才回到项目契约原文重新阅读。

## 23. 交付结论

年度 AI 总结的核心不是“每年调用一次 AI”，而是建立一份按年度沉淀、由事实指纹驱动更新、可追溯且不污染事实源的运动生涯报告。

最终规则应保持简单：

```text
年度卡片永远可以查看。
没有报告时可以生成。
年度事实变化时可以更新。
事实没有变化时只读取沉淀结果。
AI 只写叙事，不写事实。
```
