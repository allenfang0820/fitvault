# RCV2-08 工程级执行提示词

任务：测试矩阵、真实数据与发布门禁冻结

目标：在编码前冻结 Records Center V2 每一阶段的测试范围、真实数据策略、截图验收、平台验收和发布授权边界，作为后续任务选择验证命令和升级回归的依据。

输入摘要：

- `RCV2-03` 至 `RCV2-07` 已冻结 Registry、质量、schema/cache/route、API/ViewModel、视觉/交互。
- Golden fixtures 已覆盖骑行功率、徒步海拔、泳池、公开水域、越野路线边界。
- 真实库只读审计显示：普通骑行 94、电助力 2、徒步 7、登山 5、步行 38、游泳 2 且均公开水域、越野 0。
- 用户明确要求：新运动真实数据先全部保持候选，不写入真实库；暂时不要打包。

前置依赖：`RCV2-04`、`RCV2-05`、`RCV2-06`、`RCV2-07`。

文件范围：

- 可写：本提示词、`docs/records_center_v2_rcv2_08_test_release_gate_contract.md`、完成报告、滚动摘要、V2 任务清单。
- 禁止：业务代码、测试代码、API contract JSON、前端、真实库、打包产物。

冻结契约：

- 后续任务必须优先选择本文矩阵中的定向测试；高风险变更升级到宽回归。
- Fixture 通过不等于真实数据 Verified。
- `RCV2-40` 前真实数据只能备份、staging 和 dry-run。
- 没有用户明确授权，不得 apply 真实库，不得打包、签名、公证或替换发布包。
- macOS、Windows 与最终打包是独立门禁。

实施步骤：

1. 建立阶段测试矩阵：Registry、schema、Resolver、API、前端、集成、安全、平台。
2. 冻结 golden fixture 使用策略和扩展规则。
3. 冻结真实数据 dry-run、staging、人工抽样和 candidate-only 验收规则。
4. 冻结性能目标、安全扫描、日志和观测证据。
5. 冻结 macOS、Windows、打包授权和发布检查表。
6. 明确后续每个任务建议验证命令和升级条件。
7. 更新滚动摘要和任务状态。

非目标：

- 不新增测试代码。
- 不运行真实库 migration。
- 不打包。
- 不修改业务实现。

验证：

- 文档检查：任务范围覆盖 RCV2-09 至 RCV2-44、真实库禁止 apply、打包禁止、fixture 非真实验收、平台独立门禁。
- 运行 golden fixture 测试作为 Milestone A 末尾轻回归。

完成定义：

- 后续每个任务都能从本文直接选择定向测试、升级回归范围和发布/真实数据门禁。
